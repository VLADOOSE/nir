# Дизайн: сравнение предложений поставщиков (часть 3a)

- **Дата:** 2026-07-09
- **Статус:** согласовано (brainstorming), готов к writing-plans
- **Рынок:** оба (механика рынко-нейтральна)
- **Тема-родитель:** «разбор ответов КП» (часть 3; делится на 3a сравнение + 3b авто-парс цены — отдельными спеками)

## 1. Контекст и проблема

Оператор запросил КП у нескольких поставщиков (`PriceRequest` per-поставщик, `items` = позиции с `responsePrice`). Цены ответов сейчас **вводятся вручную** в «Калькулятор наценки» карточки КП (кнопка «Сохранить ответы»). Но сравнить предложения между поставщиками негде — карточки КП идут по одной, side-by-side нет. Нужна сводная таблица «какой поставщик по какому лоту дешевле».

## 2. Границы

- **В ЭТОМ спеке (3a):** read-only сводка по тендеру — матрица лоты × поставщики, цены ответов, подсветка минимума по лоту, итоги по поставщикам.
- **НЕ здесь:** авто-парс цены из письма (**3b**, отдельный спек); выбор победителя по лоту / сплит-заказ (follow-up); сводное КП клиенту (отдельная тема бэклога «КП-генератор»); наценка пер-поставщик (единый контрол).

### Что НЕ трогаем (уже есть)
- `PriceRequest` / `PriceRequestItem.responsePrice` (ручной ввод + `PUT /{id}/responses`).
- Калькулятор наценки в карточке КП (`_markup`, дефолт 25%).
- Приём ответов (`MailReceiveService` → RESPONDED, тело в `pr.note`).

## 3. Согласованные решения (brainstorming)

| Решение | Выбор |
|---|---|
| Что позволяет | **показ + лучшая (мин.) цена по лоту**, итоги; без выбора победителя |
| Где логика | **бэкенд-эндпоинт** (пивот/мин/итоги на Java, тестируемо) |
| Где UI | **отдельный standalone-компонент-модалка** (не секция карточки — не трогать 16 kB бюджет) |
| Что сравниваем | **сырые цены ответов** (мин по лоту); наценка — единый контрол в модалке |

## 4. Бэкенд — эндпоинт + сервис

`GET /api/tenders/{id}/offer-comparison` → `OfferComparisonResponse`:
```
OfferComparisonResponse {
  List<Lot> lots;                 // лоты тендера, по которым есть хоть один КП-item
  List<Supplier> suppliers;       // поставщики (PriceRequest) этого тендера
  List<Cell> cells;               // ответы по (lot, supplier)
  Map<Long,Long> bestByLot;       // lotId → priceRequestId с мин. responsePrice (пропущен, если ответов по лоту нет)
  Map<Long,BigDecimal> totalsBySupplier; // priceRequestId → Σ(responsePrice × quantity) по отвеченным позициям
}
Lot { Long lotId; Integer lotNumber; String lotName; Integer quantity; }
Supplier { Long priceRequestId; String distributorName; String status; }
Cell { Long lotId; Long priceRequestId; BigDecimal responsePrice; Integer quantity; }
```
- **`OfferComparisonService.build(tenderId)`** (`@Transactional(readOnly=true)` или под OSIV): грузит `PriceRequest`-ы тендера + items.
  - **`suppliers`** = только PR, у которых есть **≥1 позиция с непустой `responsePrice`** (ответившие ценой; SENT-без-цен в матрицу не тянем — иначе пустые столбцы).
  - **`lots`** = только лоты, по которым есть **≥1 ячейка с ценой** (иначе пустые строки).
  - **`cells`** = responsePrice per (lot × supplier) среди отобранных; отсутствие — ячейки нет (фронт рисует «—»).
  - **`bestByLot`** = PR с минимальной `responsePrice` по лоту (тай-брейк — меньший `priceRequestId`; лот без единого ответа в карту не попадает — но такие лоты уже отфильтрованы).
  - **`totalsBySupplier`** = Σ(`responsePrice × quantity`) по позициям поставщика с непустой ценой.
- **Гард рынка:** тендер через `tenderService.findById` (== `em.find`, минует фильтр) → `requireCurrentMarket(tender.getMarket())` → `NotFoundException` (эталон — `LotSourcingService`/`setProposedEquipment`).
- Только чтение; схема БД не меняется.
- Контроллер: метод в `TenderController` рядом с `lot-sourcing` (тот же паттерн), без `@PreAuthorize` (чтение, как lot-sourcing).

## 5. Фронт — компонент-модалка `app-offer-comparison`

Новый standalone-компонент `frontend/src/app/pages/tenders/offer-comparison.component.ts` (**свой `styles: []` бюджет — не трогает 16 kB карточки**; по образцу `app-bulk-price-modal`):
- `@Input tenderId`, `(close)` — открывается кнопкой; грузит `GET /offer-comparison`.
- Таблица: строки — лоты (№, название, кол-во), столбцы — поставщики; в ячейке цена ответа (или «—»). **Мин. цена по лоту** — зелёная подсветка (`bestByLot`). Строка «Итого» = `totalsBySupplier` по каждому поставщику. Пусто → мягкая заглушка «Нет ответов для сравнения».
- **Единый контрол наценки %** (инпут, дефолт 25) — рядом с ценой показывает «+маржа»/«с наценкой» (как калькулятор карточки), пересчёт клиентски.
- Кнопка **«Сравнить предложения»** в секции «Запросы КП» карточки тендера — показывается при **≥2 КП, у которых есть хотя бы одна введённая цена ответа** (`items` c непустым `responsePrice`; совпадает с фильтром DTO `suppliers` — модалка не откроется пустой). Данные для условия уже в `priceRequests` карточки. Клик → модалка.
- `ApiService.getOfferComparison(tenderId)`; паттерны `NotificationService`, `cdr.detectChanges()`.

## 6. Тестирование

- **`OfferComparisonServiceTest`** (`@SpringBootTest @Transactional`): тендер с 2 поставщиками и пересекающимися лотами → `bestByLot` = дешёвый PR по каждому лоту (в т.ч. тай-брейк равных цен → меньший prId); `totalsBySupplier` = Σ(цена×кол-во); лот, по которому один поставщик не ответил (null-cell) — не мешает мину; тендер без ответов → пустые lots/cells; **гард чужого рынка** → `NotFoundException`.
- **Фронт-гейт:** `cd frontend && npm run build` (новый компонент — свой бюджет; карточка тендера +кнопка +1 тег `<app-offer-comparison>`).
- **Живая проверка (Playwright):** KZ-тендер с ≥2 КП, у которых введены `responsePrice` по лотам → «Сравнить предложения» → модалка: матрица корректна, мин по каждому лоту подсвечен зелёным, итоги совпадают; смена наценки пересчитывает «с наценкой».
- **Гейт «зелёного»:** только 2 пред-существующих `ApplyAutoFillServiceTest`.
