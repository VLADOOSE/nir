# Умный реестр-матч по лоту + габариты-v2 — дизайн

**Дата:** 2026-07-04. **Ветка:** `feat/smart-registry-lot-match`.
**Статус:** одобрен пользователем (брейнсторм 2026-07-04, «ок, главное сделай»).

## 1. Проблема

Кнопка «Реестр» у лота (`candidatesForLot`) матчит триграммой (`word_similarity <%`, порог 0.6)
по канцелярскому названию лота. Живые замеры:
- «Устройство оцифровки рентген снимков» при пороге 0.6 → **0 кандидатов** (длинная фраза целиком
  за порогом), хотя в реестре есть «Оцифровщик рентгеновских изображений REGIUS SIGMA 2» и сканеры VistaScan.
- Понижение порога до 0.3 → топ засоряется по слову «Устройство» («Устройство регистрации
  отоакустической эмиссии»…), REGIUS тонет на 6-м месте.
- «Первые N слов» — хрупко (граница N между 2 и 3 непредсказуема) — откатили.
- Полнотекстовый `ts_rank` **дисквалифицирован замером**: русский стеммер не склеивает
  `оцифровки(оцифровк) ≠ оцифровщик`, `рентген(рентг) ≠ рентгеновских(рентгеновск)` — запрос не найдёт цель.
  Триграммы эту морфологию склеивают.
- LLM — пользователь отклонил (без ключей/оплаты, детерминированно).

Валидированное направление (живой замер): **стоп-слова канцелярита + пословный триграммный скоринг
с покрытием всех значимых токенов** — REGIUS и «Система цифровая для… интраоральных рентгеновских
снимков» выходят в топ. Нюанс: нужен позиционный вес (головное существительное первым), иначе
одиночные совпадения («портативный») разбавляют топ.

Вторая жалоба: габариты из ТЗ берутся только триплетом «A×B×C» — поосевые («длина не более X мм»)
и двумерные («размеры 55*80 мм») пропускаются.

## 2. Дизайн

### 2.1 `LotQueryTokenizer` (util, чистая функция)

`record WeightedToken(String token, double weight)`; `static List<WeightedToken> tokenize(String name, String specCharacteristics)`.

- Нормализация: lower; сплит по не-буквам, дефисные слова целиком («дефибриллятор-монитор» — один токен).
- Стоп-слова (два списка-константы):
  - канцелярит: устройство, аппарат, аппаратный, система, комплекс, изделие, прибор, оборудование,
    комплект, набор, медицинский/-ая/-ое/-ие, товар, штука;
  - служебные: для, и, с, со, по, на, из, к, от, до, в, или, не, более, менее.
- Отбрасываем токены короче 3 букв и чисто числовые.
- Из названия лота — до **5** токенов, позиционные веса `1.0, 0.7, 0.5, 0.4, 0.3`.
- Из `specCharacteristics` (блок «характеристики закупаемых товаров» разобранного ТЗ,
  `TechSpecExtractor.characteristics`) — ещё до **5** токенов с весом **×0.5** от той же лестницы,
  без дублей с именными токенами. Это чинит расходники («Электрод» + ТЗ → «пластинки, электрофореза,
  элэскулап») и добавляет сигнальные слова («детектор»).
- Пусто после фильтров → пустой список (вызывающий уходит в фолбэк).

### 2.2 `MedRegistryRepository.searchByTokens` (нативный запрос)

Токены и веса передаются строками с разделителем `|` (`string_to_array`), чтобы не возиться
с массивами в Hibernate:

```sql
WITH q AS (
  SELECT t.tok, w.wgt::float8
  FROM unnest(string_to_array(:tokens,'|')) WITH ORDINALITY AS t(tok, i)
  JOIN unnest(string_to_array(:weights,'|')) WITH ORDINALITY AS w(wgt, j) ON i = j
)
SELECT * FROM (
  SELECT m.*, (SELECT sum(q.wgt * word_similarity(q.tok, m.name)) / sum(q.wgt) FROM q) AS score
  FROM med_registry m
  WHERE EXISTS (SELECT 1 FROM q WHERE q.tok <% m.name)   -- GIN-индекс, порог 0.6 не трогаем
) s WHERE s.score >= 0.2
ORDER BY s.score DESC LIMIT :limit
```

(точная форма — подзапрос/LATERAL по вкусу реализации; ключевые свойства: `<%` для индекса,
ранг = взвешенное покрытие ВСЕХ токенов, отсечка `score >= 0.2` во внешней обёртке).

### 2.3 `RegistryMatchService.candidatesForLot`

- `lot.manufact` непуст (бренд задан — ручные/частные лоты) → старый путь `findCandidates(equipName, manufact)`.
- Иначе: `tokenize(equipName, characteristics(requiredSpec))` → пусто → фолбэк `findCandidates`
  (как сейчас); непусто → `searchByTokens`.
- `findCandidates` (оборудование, частные заявки, сверка) — НЕ трогаем.
- DTO/UI без изменений: тот же `RegistryCandidateResponse.score` (0..1), панель «Реестр» у лота.

### 2.4 Габариты-v2 (`SpecConstraintExtractor`)

Приоритет источников (первый сработавший или их комбинация по осям):
1. **Триплет** `A×B×C (мм|см|м)` с ключевым словом — как сейчас (без изменений).
2. **Поосевые**: `(длина|глубина) … (не более|до|≤|максимум) X (мм|см|м)` → `maxLengthMm`;
   `ширина …` → `maxWidthMm`; `высота …` → `maxHeightMm`. Каждая ось независимо; «не менее/от/минимум» игнор.
3. **Двумерный**: `(размер\w*|габарит\w*) … A[х×*]B (мм|см|м)` → length+width (height null).
Вес — без изменений. Все значения нормализуются в мм. Триплет, если найден, перекрывает поосевые/двумерный.
Питает и `scoreLot` (specDerived), и кнопку «ТЗ» (запись в лот) автоматически.

## 3. Тестирование

- **Юниты** `LotQueryTokenizer`: канцелярит/служебные выброшены, дефисные целиком, веса позиционные,
  ТЗ-токены ×0.5 без дублей, пусто → пусто.
- **Юниты** `SpecConstraintExtractor` v2: поосевые (все три оси, вперемешку с «не менее»),
  двумерный с ключевым словом и без (без — игнор), приоритет триплета, см/м нормализация.
- **Золотой набор** (интеграционные, живой реестр ~14k в nirdb, стиль `RegistryLotMatchTest`):
  1. «Устройство оцифровки рентген снимков» → в топ-5 есть имя с «оцифровщик» или «рентгеновских снимков»;
  2. «Дефибриллятор-монитор бифазный портативный» → топ-3 содержит «дефибриллятор»;
  3. «Пульсоксиметр» → топ содержит «пульсоксиметр» (короткое имя не деградировало);
  4. «Электрод» + requiredSpec с характеристиками «Резиновые пластинки для аппарата электрофореза
     "Элэскулап"» → топ-5 содержит «электрофорез» или «ЭЛЭСКУЛАП» (обогащение из ТЗ работает);
  5. лот с manufact (бренд) → идёт старым путём (поведение не изменилось).
- **Гейты:** `./gradlew test` (только 2 известных `ApplyAutoFillServiceTest`), `npm run build`
  (фронт не меняется — smoke).
- **Live (Playwright):** `/tenders?openId=852` → «Реестр» у лота показывает оцифровщики/сканеры
  рентген-пластин; электрод-лот (тендер 17279420-1) после «ТЗ» → осмысленные кандидаты.
