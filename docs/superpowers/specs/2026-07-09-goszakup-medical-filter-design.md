# Дизайн: точный фильтр импорта goszakup (медтовары, без услуг)

- **Дата:** 2026-07-09
- **Статус:** согласовано (brainstorming), готов к writing-plans
- **Рынок:** KZ (импорт goszakup)
- **Мотив:** после «Обновить тендеры» в активных ~39 KZ-тендеров, и почти все — мусор: дроны («аппарат летательный»), «Аппарат акима», медотходы, медосмотр, пошив медхалатов, обучение медперсонала. Реального медоборудования не найти.

## 1. Контекст и проблема

Импорт (`GoszakupImportService`) тянет ленту goszakup `/trd-buy` (новейшие-первыми, окно `since-days=30`, до `max-pages=60` × `page-size=50` ≈ 3000 объявлений) и фильтрует **keyword-списком по имени объявления** (`name_ru`, ~25 слов). Серверного поиска у goszakup НЕТ (v2 фильтры игнорирует, v3 `nameRu` — только точное совпадение), поэтому фильтр локальный.

Две беды текущего фильтра (подтверждены на реальных данных):
1. **Ловит мусор.** «аппарат» → дроны, «Аппарат акима»; «медицинск» → услуги (медотходы, медосмотр, пошив халатов, обучение). Из 39 «активных» реального медоборудования почти нет.
2. **Пропускает реальное.** Тендер на медтехнику с обобщённым именем объявления в keyword-список не попадает и отсекается, хотя его *лоты* — чистая медтехника.

В `TrdBuyDto` категории/ОКЭД нет (`name_ru`, `count_lots`, статус, БИНы, даты). Но у **лотов** (`LotDto`) есть `name_ru`/`description_ru` — реальные названия позиций. Судить о «медицинскости» надо по лотам, а не по имени объявления.

## 2. Границы

- **В спеке:** двухступенчатый отбор в импорте — чистка keyword-предфильтра (ступень 1) + новая проверка релевантности по лотам (ступень 2, `MedicalRelevanceFilter`).
- **НЕ здесь:** deep-скан всей ленты по лотам (отклонён — тысячи доп. запросов); чистка уже налитых данных (**fix-forward**, решение оператора: старое не трогаем, само истечёт/переимпортируется); обучаемость словаря (термины хардкодим); UI.

### Согласованные решения (brainstorming)
| Решение | Выбор |
|---|---|
| Подход | **фильтр по лотам (ступень 2) + чистка keyword-слов (ступень 1)** |
| Что релевантно | **медицинские ТОВАРЫ (аппараты + расходка), услуги отсеиваем** |
| Существующие данные | **только вперёд** (не чистим; мусорные активные истекут по дедлайну) |
| Ступень 1 (предфильтр) | оставляем (дешёвая сеть кандидатов; без неё — качать лоты для 3000 объявлений) |
| Списки терминов | **захардкодить** в фильтре (обучаемость — потом) |

## 3. Двухступенчатый отбор

**Ступень 1 — keyword-предфильтр по имени объявления (как сейчас, но чище).** Грубая дешёвая сеть кандидатов: из ~3000 объявлений оставляет те, чьё `name_ru` содержит слово из списка. Правки:
- убрать мусор-ловушки: голое **«аппарат»** (дроны, «аппарат акима») — покрытие реальных девайсов остаётся через специфичные слова («ивл», «узи», «рентген», «томограф»…);
- добавить медтоварные термины, чтобы ловить больше кандидатов (напр. «изделия медицинского назначения», «расходн», «перчат», «шприц», «хирургическ»).
- Список — дефолт `GOSZAKUP_KEYWORDS` в `application.yaml`.
- **Роль неизменна:** это только отбор кандидатов; точность даёт ступень 2. Обобщённо-названные объявления (без keyword в имени) по-прежнему пропускаются — это принятый компромисс (deep-скан отклонён).

**Ступень 2 — проверка релевантности по лотам (НОВОЕ, точная).** Кандидат в `importOne` **и так уже качает** subject+lots (`client.fetchLots`). После загрузки — `MedicalRelevanceFilter.isRelevant(name, lotTexts)`. Не прошёл → тендер НЕ создаётся (skip). **Без лишних API-запросов** сверх тех, что и так делались для кандидатов.

## 4. `MedicalRelevanceFilter` (новый util, самодостаточный, тестируемый)

Пакет `integration/goszakup/` (goszakup-специфичен). Чистая сигнатура на строках (без DTO — тестируемо):
```java
public static boolean isRelevant(String announcementName, List<String> lotTexts)
```
где `lotTexts` = для каждого лота `name_ru + " " + description_ru` (собирает вызывающий из `List<LotDto>`).

**Наборы терминов (хардкод, lower-case; черновой — финализируется в реализации под голден-тест):**
- **POSITIVE (медтовар):** виды МИ (`узи`, `ультразвук`, `рентген`, `флюорограф`, `маммограф`, `томограф`, `мрт`, `кт`, `ивл`, `вентил`, `наркозн`, `анализатор`, `гематологич`, `биохимич`, `центрифуг`, `микроскоп`, `стерилизатор`, `автоклав`, `эндоскоп`, `гастроскоп`, `колоноскоп`, `дефибриллятор`, `монитор пациента`, `кардиограф`, `экг`, `спирометр`, `инкубатор`, `облучатель`, `физиотерап`, `дефибрил`) + товары/расходка (`перчат`, `шприц`, `катетер`, `зонд`, `бинт`, `пластыр`, `электрод`, `реагент`, `тест-систем`, `изделие медицинск`, `медицинского назначения`, `расходн`, `хирургическ`, `имплант`, `протез`, `шовн`, `игл`).
- **NEGATIVE (услуга/не-медицина) → лот НЕ товар:** `услуг`, `работы по`, `работ ` , `обучен`, `осмотр`, `утилизац`, `удаление`, `отход`, `ремонт`, `монтаж`, `обслуживан`, `замер`, `аренд`, `страхован`, `питани`, `охран`, `уборк`, `пошив`, `летательн`, `беспилотн`, `дрон`, `потолок`.

**Правило:** лот считается «медтоваром», если его текст содержит POSITIVE **и не** содержит NEGATIVE. Тендер релевантен, если **≥1 лот — медтовар**. (Один медтоварный лот тянет весь тендер — напр. тендер «Перчатки + Покрывало» остаётся ради «Перчаток».)

**Фолбэк при пустых лотах** (сеть/404): судим по `announcementName` — специфичный POSITIVE-девайс без NEGATIVE → оставить; иначе (только услуга/дженерик) → дроп. Чтобы не терять реальное на транзиентных ошибках, но и не плодить мусор.

**Проверка на реальных примерах (голден):**
| Текст | POS | NEG | Итог |
|---|---|---|---|
| «Приобретения аппарат летательный (дрон)» | — | летательн/дрон | ДРОП |
| «Аппарат акима сельского округа» | — | — | ДРОП (нет POS) |
| «удаление медицинских отходов класса Б» | — | удаление/отход | ДРОП |
| «услуги планового медицинского осмотра» | — | услуг/осмотр | ДРОП |
| «пошив медицинских халатов» | — | пошив | ДРОП |
| «обучение среднего медицинского персонала» | — | обучен | ДРОП |
| «УЗИ-сканер Mindray DC-70» | узи | — | ОСТАВИТЬ |
| «Аппарат ИВЛ реанимационный» | ивл | — | ОСТАВИТЬ |
| «Перчатки нитриловые стерильные» | перчат | — | ОСТАВИТЬ |
| «Реагенты для гематологического анализатора» | реагент/анализатор | — | ОСТАВИТЬ |

## 5. Куда встраивается

- **`GoszakupImportService.importOne`** — единственная точка: после `List<LotDto> lots = client.fetchLots(...)` и перед `writer.upsertOne(...)` добавить:
  ```java
  List<String> lotTexts = lots.stream()
      .map(l -> (safe(l.getNameRu()) + " " + safe(l.getDescriptionRu())).trim()).toList();
  if (!MedicalRelevanceFilter.isRelevant(d.getNameRu(), lotTexts)) {
      sum.setSkipped(sum.getSkipped() + 1);
      return; // тендер не пишем
  }
  ```
- Инъекция не нужна (статический util). `matched` считается в `processItems` до сети (= «прошёл предфильтр имени»); ступень-2 дропы идут в `skipped` — расхождение `matched > created+updated` теперь отражает работу лот-фильтра (полезно для наблюдаемости; отметить в summary-логе).
- **Ступень 1** — правка дефолта `GOSZAKUP_KEYWORDS` в `application.yaml`.

## 6. Тестирование

- **`MedicalRelevanceFilterTest`** — голден-набор из §4 (реальный мусор → ДРОП; медтовары → ОСТАВИТЬ) + тендер-уровень (≥1 медтоварный лот → релевантен; все услуги → дроп) + фолбэк пустых лотов.
- **`GoszakupImportService` тест** (или на `importOne`/`processItems` с моком клиента): кандидат, чьи лоты — услуги, не создаётся (`skipped++`, `created` не растёт); кандидат с медтоваром — создаётся. (Существующие импорт-тесты не ломать.)
- **Нет регресса охвата ступени 1:** тест/проверка, что реальные девайс-названия по-прежнему проходят предфильтр после удаления голого «аппарат» (напр. «Аппарат ИВЛ»→`ивл`, «Аппарат УЗИ»→`узи`, «Рентген-аппарат»→`рентген`, «Наркозно-дыхательный аппарат»→`наркоз`). Специфичное слово девайса должно быть в списке ступени 1.
- **Гейт:** только 2 пред-существующих `ApplyAutoFillServiceTest`. Фронт не трогаем.
- **Живая проверка (токен есть):** `GOSZAKUP_TOKEN=$(cat /tmp/goszakup.token) ./gradlew bootRun`, кнопка «Обновить тендеры» на `/tenders` (KZ) → в summary виден рост `skipped`; в активных больше нет дронов/медотходов/медосмотра, видны реальные медтовары. (Старый мусор остаётся до истечения дедлайна — fix-forward.)
