# Дизайн: Переключатель рынков (Регион-Мед РФ ↔ West-Med KZ) — Блок A

**Дата:** 2026-06-24
**Статус:** утверждён, готов к плану
**Контекст:** первый блок расширения АИС в мульти-рыночный продукт (реализует отложенное решение #10 из реестр-спеки). Часть roadmap «поток Частники + мульти-рынок» из `HANDOFF-АИС-медоборудование.md`.

---

## 1. Цель

Сделать АИС двух-«вселенной»: **Регион-Мед (РФ, ₽)** и **West-Med (KZ, ₸)** — две отдельные компании со своими данными (каталог/тендеры/заказчики/поставщики/заявки раздельны). Глобальный переключатель в шапке выбирает активный рынок; весь интерфейс показывает только данные этого рынка, в его валюте, с его идентичностью компании.

Заказчик: данные логически разнесены колонкой `market` (пользователю колонка не видна — он просто переключает рынок и работает в нём). Бизнес-модель (раздельные компании, РФ→KZ реэкспорт, Регион-Мед = канал сорсинга для West-Med) — см. реестр-спеку §2.

## 2. Журнал решений

| # | Развилка | Решение |
|---|----------|---------|
| 1 | Разделение данных | **Общая БД + колонка `market`** на «вселенных» таблицах (не отдельные схемы/БД) |
| 2 | Что строим первым | **Блок A — переключатель рынков** (фундамент), частники (B) — после |
| 3 | Модель рынка | enum из двух значений `RF`/`KZ` (отдельной таблицы не нужно) |
| 4 | Скоупинг чтения | **Hibernate `@Filter`**, включается per-request из заголовка `X-Market` (одно место, скоупит все JPQL/criteria) |
| 5 | Штамповка записи | **Явно** в сервисах: при создании `entity.setMarket(MarketContext.get())` |
| 6 | Передача активного рынка | HTTP-заголовок `X-Market: RF\|KZ` (фронт-интерсептор → бэк servlet-filter → ThreadLocal) |
| 7 | Валюта | Производная от рынка: RF→₽ RUB, KZ→₸ KZT. Символ из активного рынка |

## 3. Модель рынка

Enum `Market` (бэк) и тип на фронте:
- `RF` — компания «Регион-Мед», валюта `RUB` (₽)
- `KZ` — компания «West-Med», валюта `KZT` (₸)

Метаданные рынка (название компании, валюта, символ) — в одном месте: enum на бэке + `MarketService` на фронте.

---

## 4. Модель данных

**Скоупятся по рынку** (новая колонка `market VARCHAR(2) NOT NULL DEFAULT 'RF'`):
`tender`, `facility`, `distributor`, `med_equipment`, `activity_apply`, `price_request`.

Существующие данные (самарские RF-тендеры и каталог) автоматически получают `RF` через DEFAULT — текущая БД не ломается.

**Остаются общими** (без колонки):
- `equipment_type` — типы оборудования (УЗИ/ИВЛ) универсальны для обоих рынков.
- `user_account` — оператор работает в обеих компаниях, переключается свободно.
- `med_registry` — это KZ-реестр НЦЭЛС; релевантен только при активном рынке KZ. Для RF позже будет отдельный источник (Росздравнадзор) — вне этого блока. Реестр-функции (сверка) при активном RF показываем как «доступно для рынка KZ» либо просто пустой результат.

**Индексы:** на каждую скоупленную таблицу добавить индекс по `market` (частый фильтр).

`price_request_item` / `apply_item` — НЕ получают колонку (наследуют рынок через родителя `price_request`/`activity_apply`).

## 5. Бэкенд

**`Market` enum** (`com.vladoose.nir.entity`): `RF`, `KZ` + методы `currencyCode()` (RUB/KZT), `currencySymbol()` (₽/₸), `companyName()`.

**`MarketContext`** — ThreadLocal-холдер активного рынка на текущий запрос: `get()`, `set(Market)`, `clear()`. Дефолт `RF`, если заголовок отсутствует/некорректен.

**`MarketFilter`** (servlet `OncePerRequestFilter`): читает заголовок `X-Market`, кладёт в `MarketContext`, включает Hibernate-фильтр `marketFilter` на текущей сессии (`entityManager.unwrap(Session.class).enableFilter("marketFilter").setParameter("market", market.name())`), в `finally` — `MarketContext.clear()`.

**Hibernate `@Filter` на скоупленных entity:**
```java
@FilterDef(name = "marketFilter", parameters = @ParamDef(name = "market", type = String.class))
@Filter(name = "marketFilter", condition = "market = :market")
```
(`@FilterDef` — один раз, например на `package-info` или на любой entity; `@Filter` — на каждой из 6 скоупленных.) Фильтр скоупит все JPQL/criteria-чтения автоматически.
> ⚠️ Hibernate `@Filter` НЕ применяется к `EntityManager.find()`/`getReference()` (только к queries) и к native-запросам. Для MVP это приемлемо: списки/поиск/дашборд идут через JPQL → скоупятся; доступ по прямому id из другого рынка (редкий внутренний кейс) — деферим как hardening. Native-запрос реестра (`med_registry`) — общая таблица, фильтр не нужен.

**Штамповка при создании:** entity-поле `market` (enum, `@Enumerated(STRING)`, `@Column(nullable=false)`). В create-методах сервисов скоупленных сущностей: `entity.setMarket(marketContext.get())` перед `save`. (Update/delete работают по уже отфильтрованным выборкам.)

**Валюта:** при создании тендера `currency` берётся из активного рынка (поле уже есть). Сериализация сумм — без изменений; символ валюты — на фронте.

**`CompanyInfo` market-aware:** существующий `CompanyInfo`/`PdfCompanyHeader`/`ExcelCompanyHeader` — отдаёт реквизиты по активному рынку (West-Med для KZ, Регион-Мед для RF). Небольшое добавление; если данных West-Med нет — заглушка-плейсхолдер.

**Реестр-полиш (заодно, отдельные коммиты):**
- Иконка `badge-check` — уже починена.
- Матчинг: в `MedRegistryRepository.findCandidates` добавить `word_similarity`/`<%` — короткий бренд («Mindray») должен находить кандидата в длинном юр-имени («Shenzhen Mindray Bio-Medical Electronics»). Скор-формула учитывает и `similarity`, и `word_similarity` (берём максимум по производителю). Порог через `<%` (word_similarity threshold). Перетест на реальных данных.

## 6. Фронтенд

**`MarketService`** (`providedIn: 'root'`): активный рынок (signal/BehaviorSubject), дефолт `RF`, персист в `localStorage`. Отдаёт `{ market, currencySymbol, companyLabel }`. Метод `setMarket(m)` → меняет, перезагружает данные (через событие/навигацию), пишет в localStorage.

**Селектор в шапке** (`layout.component.ts`, рядом с логотипом «РМ»): переключатель «Регион-Мед (РФ) ₽» ↔ «West-Med (KZ) ₸». Визуально заметный (pill/dropdown). Смена → `MarketService.setMarket` → перезагрузка текущей страницы/данных.

**HTTP-интерсептор** `marketInterceptor`: добавляет заголовок `X-Market: <active>` к каждому `/api`-запросу (рядом с `authInterceptor` в `app.config.ts`).

**Валюта в UI:** заменить захардкоженные `₽`/`&#8381;` на символ из `MarketService` (пайп `marketCurrency` или геттер). Затрагивает: тендеры, оборудование, заявки, отчёты, КП, детальные карточки.

**Логотип/идентичность:** «РМ» / «АИС Регион-Мед» в шапке → меняется на «WM» / «АИС West-Med» по активному рынку.

## 8. Скоуп блока A

**В скоупе:**
- enum `Market`, колонка `market` на 6 таблицах + индексы, DEFAULT 'RF'.
- `MarketContext` + `MarketFilter` + Hibernate `@Filter` (скоупинг чтения).
- Штамповка `market` при создании в сервисах 6 сущностей.
- Фронт: `MarketService`, селектор в шапке, `marketInterceptor`, валюта по рынку, идентичность в шапке.
- `CompanyInfo` market-aware (реквизиты компании по рынку).
- Демо-данные West-Med/KZ.
- Реестр-полиш: иконка (готово) + `word_similarity`-матчинг.

**Вне скоупа (блоки B+):**
- Частные заявки (раздел, карточка, ручной ввод).
- Автопарсинг почты (IMAP, Excel/корзина).
- Запрос КП по группам поставщиков + справочник «кто что возит/имеет РУ».
- RF-реестр (Росздравнадзор).
- Hardening: запрет доступа по прямому id из чужого рынка (`@Filter` не покрывает `find()`).
- Полноценные реквизиты West-Med в PDF/Excel (если нет данных — плейсхолдер).

## 9. Обработка ошибок / edge

- Заголовок `X-Market` отсутствует/некорректен → дефолт `RF` (бэк не падает).
- Hibernate-фильтр включается per-request и не «протекает» между запросами (ThreadLocal чистится в `finally`).
- `med_equipment.market` у новых позиций KZ — реестр-сверка работает (med_registry общий), бейджи НДС — как раньше.
- При активном RF реестр-сверка показывает «реестр доступен для рынка KZ» (или пустой список) — не ломается.

---

## 10. Тестирование

- **Unit/integration (бэк):** создание тендера/оборудования при `MarketContext=KZ` штампует `KZ`; список при фильтре `KZ` не возвращает RF-записи и наоборот (Hibernate-фильтр); дефолт `RF` при отсутствии заголовка. `Market.currencySymbol()`/`currencyCode()`.
- **Матчинг:** `findCandidates('Mindray', ...)` теперь возвращает «Shenzhen Mindray…» (word_similarity) — на реальном дампе.
- **Фронт:** сборка `npm run build`; ручной e2e — переключение рынка в шапке меняет список тендеров/каталог, валюту (₽↔₸) и идентичность; данные RF не видны при KZ.
- **E2E-смоук:** старт → RF по умолчанию (самарские тендеры, ₽) → переключить на KZ → видны только KZ-демо-данные в ₸ → обратно на RF.
