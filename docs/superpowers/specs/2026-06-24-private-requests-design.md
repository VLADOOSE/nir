# Дизайн: Частные заявки (Блок B)

**Дата:** 2026-06-24
**Статус:** утверждён, готов к плану
**Контекст:** второй блок расширения АИС (после переключателя рынков). Реализует поток «Частники» из `HANDOFF-АИС-медоборудование.md` — заявки от частных клиник на медоборудование.

---

## 1. Цель

Добавить раздел **«Частные заявки»** рядом с тендерами: заявки от частных клиник на медоборудование. В отличие от тендеров (где по закону нельзя называть бренды → параметры → наш автоподбор), частная клиника **прямо называет бренд/модель** нужного оборудования. Поэтому автоподбор не нужен — нужна **проверка регистрации названной модели** (реестр НЦЭЛС) + сорсинг (запрос КП у поставщиков).

Поток: заявка приходит (сейчас — ручной ввод, потом — авто-парсинг почты) → карточка заявки со строками (бренд/модель/кол-во) → по каждой строке видим статус регистрации → «Запросить КП» у поставщиков → вручную вносим ответы → наполняем заявку.

---

## 2. Журнал решений

| # | Развилка | Решение |
|---|----------|---------|
| 1 | Модель данных | **A — переиспользуем `tender`** с полем `source` (PUBLIC_TENDER / PRIVATE_REQUEST), НЕ отдельная сущность. Главный выигрыш — переиспользование КП-workflow (`PriceRequest` привязан к лотам) и контейнера заявки |
| 2 | Бренд/модель | Новое nullable поле `manufact` на `tender_lot` (у тендеров null — бренды запрещены; у частных заявок — названная клиникой модель) |
| 3 | Автоподбор | **Не используется** в частных заявках (клиника называет конкретную модель). Реестр-статус строки = `findCandidates(модель, бренд)` |
| 4 | Номер/дедлайн | Частной заявке автогенерим номер `ЧЗ-<год>-NNNN`; `tender.deadline` делаем **nullable** (у частной заявки дедлайна может не быть) |
| 5 | Граница B/C | В B — запрос КП **per-поставщик** (переиспользуем готовый `PriceRequest`-поток). Группировка «КП по поставщикам, у кого есть» + справочник — в **блоке C** |
| 6 | Шов приёма | `PrivateRequestService.createFromLines(client, lines[])` — единая точка создания. Ручная форма зовёт сейчас; авто-парсер почты (блок D) — потом |

**Доменный инсайт (от заказчика):** частные клиники дают в заявке готовые бренды/модели; в тендерах по закону так нельзя. Поэтому автоподбор — только для тендеров, а для частников — проверка регистрации названной модели.

**На будущее (заказчик):** оба потока должны наполняться автоматически — частные заявки из писем (таблица МИ на почте), тендеры с площадки. Шов `createFromLines` закладываем сейчас, чтобы авто-источники (блок D) звали тот же сервис.

## 3. Скоуп блока B

**В скоупе:**
- Поле `source` на `tender` (PUBLIC_TENDER default / PRIVATE_REQUEST) + nullable `manufact` на `tender_lot` + nullable `deadline`.
- `PrivateRequestService.createFromLines(...)` — шов приёма структурированных строк (ручной ввод сейчас).
- `PrivateRequestController` (`/api/private-requests`): list/get/create/update, скоуп по рынку, фильтр source=PRIVATE_REQUEST.
- Тендерные эндпоинты фильтруют source=PUBLIC_TENDER (частные не сыпятся в «Тендеры»).
- Реестр-статус по каждой строке заявки (переиспуем `RegistryMatchService.findCandidates`).
- Frontend: раздел «Частные заявки» (список + форма создания) + карточка заявки (строки + реестр-статус + запрос КП per-поставщик + ввод ответов).

**Вне скоупа (блоки C, D):**
- Справочник «кто из поставщиков что возит/имеет» + «запрос КП по группам поставщиков, у кого есть» (блок C).
- Авто-парсинг почты для частников и площадки для тендеров (блок D) — но шов `createFromLines` готов.
- Авто-генерация финального КП клиенту (модуль КП-генератора из HANDOFF).

## 4. Модель данных (переиспользуем тендер)

**Новое поле на `tender`:**
```sql
source VARCHAR(20) NOT NULL DEFAULT 'PUBLIC_TENDER',  -- PUBLIC_TENDER | PRIVATE_REQUEST
```
Существующие тендеры → `PUBLIC_TENDER` через DEFAULT (БД не ломается).

**Изменение `tender.deadline`:** `DATE` → **nullable** (частной заявке дедлайн не обязателен). Существующие данные не страдают.

**Новое поле на `tender_lot`:**
```sql
manufact VARCHAR(255),  -- бренд/модель для частных заявок; у тендеров NULL
```

**Java-enum** `Source { PUBLIC_TENDER, PRIVATE_REQUEST }` (`com.vladoose.nir.entity`).

**Автономер частной заявки:** `ЧЗ-<год>-<NNNN>` (порядковый в рамках года и рынка) — генерится в `PrivateRequestService` при создании (макс существующий номер + 1, либо count-based).

Переиспользуем без изменений: `facility` (= клиент-клиника), `PriceRequest`/`PriceRequestItem` (КП-workflow привязан к `tender_lot`), `ActivityApply`/`ApplyItem` (наполнение), `RegistryMatchService` (реестр), market-`@Filter` (рыночный скоупинг — `tender` уже скоупится).

## 5. Backend-компоненты

- **`Source` enum** на `Tender` (`@Enumerated(STRING)`, `@Column(nullable=false, length=20)`, `@Builder.Default = PUBLIC_TENDER`).
- **`Tender.deadline`** → nullable (убрать `nullable=false` в entity + schema).
- **`TenderLot.manufact`** → новое поле (String, nullable).
- **`PrivateRequestService`:**
  - `createFromLines(PrivateRequestCreate dto) : Tender` — **шов приёма.** Создаёт `Tender(source=PRIVATE_REQUEST)` с автономером + лоты со строками (equip_name=модель, manufact=бренд, quantity). Рынок штампуется листенером.
  - `findAll() : List<Tender>` — только PRIVATE_REQUEST (рыночный скоуп — через @Filter).
  - `findById(id)`, `update(...)`.
  - `lineRegistrationStatus(lotId) : RegistryCandidateResponse list` — реестр-статус строки через `RegistryMatchService.findCandidates(lot.equipName, lot.manufact)`.
- **`PrivateRequestController`** (`/api/private-requests`): `GET` (list), `GET /{id}`, `POST` (createFromLines), `PUT /{id}`, `GET /{id}/lines` с реестр-статусом. Записи — `@PreAuthorize("hasRole('ADMIN')")` как у прочих.
- **Фильтрация source:** `TenderService.findAll()`/`searchTenders` → добавить `source=PUBLIC_TENDER` (чтобы «Тендеры» и дашборд не показывали частные заявки). `PrivateRequestService` → PRIVATE_REQUEST. (Доступ по id — `findById` — работает для обоих; КП/заявки ссылаются по id независимо от source.)
- **DTO:** `PrivateRequestCreate` (`clientFacilityId` — выбор существующего клиента-facility; новый клиент заводится через раздел «Учреждения», не инлайн; список строк `{name, manufact, quantity}`), `PrivateRequestResponse` (заявка + строки + реестр-статус), `PrivateRequestLineResponse`. Mapper переиспользует существующие где можно.

## 6. Frontend

- **Раздел «Частные заявки»** в сайдбаре (группа «Заявки» или рядом с «Тендеры»), маршрут `/private-requests`.
  - Список заявок текущего рынка (номер `ЧЗ-...`, клиент, число строк, статус, дата).
  - Форма создания: выбор существующего клиента (facility) + строки `{наименование/модель, бренд, кол-во}` (добавить/удалить строку).
- **Карточка заявки:**
  - Шапка: клиент, номер, статус.
  - Строки: модель + бренд + кол-во + **инлайн реестр-статус** (зарегистрировано/нет + НДС-бейдж — переиспуем стиль из реестр-сверки) с топ-кандидатом РУ.
  - **«Запросить КП»**: выбрать поставщика → создать `PriceRequest` по строкам (переиспуем backend КП-поток). Показ ответов поставщиков; ручной ввод цены ответа.
  - Без автоподбора и без параметрических полей (габариты/тип) — они для тендеров.
- Валюта — по активному рынку (пайп `money` из блока A). Реестр-статус работает в активном рынке (KZ — НЦЭЛС).

## 7. Обработка ошибок / edge

- Создание заявки без строк → 400 (нужна хотя бы одна строка).
- `deadline` отсутствует у частной заявки → ок (nullable); дашборд «ближайшие дедлайны» показывает только тендеры с дедлайном.
- Реестр-статус строки при активном RF — реестр НЦЭЛС только KZ → для RF показываем «реестр доступен для рынка KZ» / пустой (как в блоке A).
- Автономер: коллизия при параллельном создании — last-write-wins/повтор (для одного оператора некритично).
- Тендерные виджеты (поиск, отчёты по тендерам) не должны включать частные заявки → фильтр source=PUBLIC_TENDER.

## 8. Тестирование

- **Unit/integration (бэк):** `createFromLines` создаёт tender(source=PRIVATE_REQUEST) с автономером + лоты с брендами, штампует активный рынок; `TenderService.findAll` НЕ возвращает частные заявки и наоборот; реестр-статус строки возвращает кандидата для известной модели; nullable deadline персистится.
- **Frontend:** сборка `npm run build`; ручной e2e — создать частную заявку (клиент + строки бренд/модель) → карточка → по строке виден реестр-статус → «Запросить КП» создаёт запрос → ввод ответа.
- **E2E-смоук:** рынок KZ → раздел «Частные заявки» → создать заявку с моделью (напр. бренд из реестра) → строка показывает «Зарегистрировано»/кандидата → раздел «Тендеры» НЕ показывает эту заявку.
