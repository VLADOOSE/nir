# Разбор «Технической спецификации» лота goszakup → структура лота — дизайн

**Дата:** 2026-07-04. **Ветка:** новая, `feat/goszakup-techspec`.
**Статус:** одобрен пользователем (брейнсторм 2026-07-04).

## 1. Контекст и цель

Все ~1027 импортных KZ-лотов приходят из v3 API без типа/габаритов/веса, `description_ru` куцый
(«медицинский»). Реальное ТЗ — в приложенном к **лоту** файле «Техническая спецификация» (PDF).
Из-за этого подбор оборудования по импортным лотам не работает (после фикса `8866e42` он честно
говорит `noCriteria`, но данных всё равно нет).

Блок: **по кнопке в лоте** скачать PDF техспеки через goszakup API, извлечь русскую ТТХ-секцию,
записать её в `requiredSpec` лота и распарсенные габариты/вес — в структурные поля. После этого
существующий подбор (`scoreLot` + `SpecConstraintExtractor`) и КП-письма работают осмысленно.

Решения брейнсторма:
1. **Триггер — кнопка «ТЗ» в строке лота** (on-demand, не при импорте: 1000+ PDF за прогон дорого).
2. **Парсер — правила** (без LLM): русская секция всегда присутствует в двуязычном PDF.
3. **Запись — текст + габариты в лот** (перезапись `requiredSpec` и `max*`-полей, идемпотентно).

## 2. Разведка API (подтверждено живым токеном 2026-07-04)

- v3 GraphQL: `Lots(filter:{trdBuyNumberAnno:"17280874-1"}){ lotNumber nameRu Files{ nameRu originalName filePath } }`
  → у лота массив `Files`, техспека — элемент с `nameRu == "Техническая спецификация"`,
  `filePath` — прямой URL (`https://ows.goszakup.gov.kz/download/trd_buy_lots_list/<hash>`).
- Скачивание: GET `filePath` с `Authorization: Bearer <GOSZAKUP_TOKEN>` → 200,
  `Content-Disposition: attachment; filename="techspec_<trdBuyId>_<fileId>.pdf"`, magic `%PDF`.
  Все увиденные техспеки — PDF (~38–48 КБ).
- Содержимое PDF **двуязычное**: сначала казахская секция, затем русская
  («Приложение 2 к конкурсной документации … Техническая спецификация закупаемых товаров…»)
  с полем «Описание и требуемые функциональные, технические, качественные и эксплуатационные
  характеристики закупаемых товаров:» — там реальное ТЗ.
- Фильтр `LotsFiltersInput` содержит `trdBuyNumberAnno` — запрашиваем по `tender.sourceExtId`
  (он равен полному номеру объявления, напр. `17280874-1`).
- Связь нашего лота с goszakup-лотом: `lot_number` у импортных лотов ПУСТОЙ, матчим по
  `equipName == nameRu`. В многолотовых тендерах (до 50 лотов) встречаются дубли имён —
  берём первый лот с файлом техспеки и помечаем ответ `ambiguous: true`.

## 3. Архитектура (бэкенд)

Синхронный on-demand разбор одного файла (~40 КБ) — без очередей/асинхронности.

- **Зависимость:** `org.apache.pdfbox:pdfbox` (извлечение текста из PDF; POI не умеет PDF).
- **`GoszakupClient`** (расширение интерфейса + `GoszakupHttpClient`):
  - `LotTechSpecRef fetchLotTechSpec(String numberAnno, String lotNameRu)` — v3-запрос выше,
    матч лота по `nameRu` (trim, case-insensitive), в его `Files` ищем `nameRu == "Техническая
    спецификация"`; `record LotTechSpecRef(String filePath, String originalName, boolean ambiguous)`;
    null — если лот/файл не найден. GraphQL-алиасы в camelCase как в существующем v3-запросе.
  - `byte[] downloadFile(String url)` — GET с Bearer-токеном (переиспользует существующий
    HTTP-хелпер; тот же таймаут-профиль, что у остальных вызовов клиента).
- **`PdfTextExtractor`** (`util/`): `static String extract(byte[] pdf)` — PDFBox, весь текст;
  пустой/нечитаемый PDF → `null`.
- **`TechSpecExtractor`** (`util/`, чистая функция + юнит-тесты): из двуязычного текста вырезает
  русскую секцию — от маркера `Приложение 2` + `Техническая спецификация` (русская шапка) до конца
  документа; если русского маркера нет — возвращает весь текст как есть. Нормализация пробелов.
- **`TechSpecService`** (`service/`): оркестрация вне транзакции (сеть) + запись отдельным
  транзакционным методом (паттерн §6, как `PriceRequestWriter`):
  1. лот + гарды (рынок, `sourceExtId`);
  2. `fetchLotTechSpec` → `downloadFile` → `PdfTextExtractor` → `TechSpecExtractor`;
  3. `SpecConstraintExtractor.extract(русский текст)` → габариты/вес;
  4. `TechSpecWriter.apply(lotId, text, constraints)` (`@Transactional`, отдельный бин):
     `requiredSpec = text` — пишется всегда (перезапись); `max*`-поля перезаписываются ТОЛЬКО
     значениями, которые экстрактор нашёл (не найденное не затирает существующее в лоте).
  - Возврат: `record ParseResult(TenderLot lot, boolean specFound, boolean dimsFound,
    boolean weightFound, boolean ambiguous, String source)`.

## 4. API

`POST /api/lots/{id}/parse-techspec` (ADMIN, `TenderLotController`):
- Ответ 200: `{ lot: TenderLotResponse, specFound, dimsFound, weightFound, ambiguous, source }`.
- Гарды/ошибки:
  - лот чужого рынка → 404 (как в proposed-equipment; findById обходит фильтр рынка);
  - `tender.sourceExtId == null` (ручной тендер) → 400 «ТЗ доступно только у импортированных тендеров»;
  - токен не настроен (`!client.isConfigured()`) → 400 «Токен goszakup не настроен (GOSZAKUP_TOKEN)»;
  - лот/файл техспеки не найден на площадке → 404 «Файл „Техническая спецификация" не найден у лота»;
  - PDF не читается / пустой текст → 422 «Не удалось извлечь текст из PDF»;
  - сетевая ошибка goszakup → 502 с сообщением.
- Ошибки НЕ меняют лот. Повторный вызов — идемпотентная перезапись.

## 5. Frontend

- Кнопка **«ТЗ»** в строке лота (`tenders.component.ts`, перед «КП»), видна только на импортных
  тендерах: `isKz() && /^\d+-\d+$/.test(tenderNumber)`. Клик → disabled+«…» → тост
  «ТЗ разобрано: спека N симв., габариты ✓/—, вес ✓/—» (+ « (неоднозначный матч лота)» при
  `ambiguous`) → `loadLots()`. Ошибка → тост с сообщением бэка.
- Баннер `noCriteria` в smart-match: добавить «…или нажмите «ТЗ» у лота — система скачает и
  разберёт техспецификацию с goszakup».
- Существующая разворачиваемая ячейка «Спецификация» показывает полный записанный текст.

## 6. Тестирование

- **Юнит:** `TechSpecExtractor` — двуязычный текст → только русская секция; текст без маркеров →
  как есть; null/пустой → null. `PdfTextExtractor` — PDF, сгенерированный PDFBox'ом в тесте →
  текст; мусорные байты → null.
- **Интеграционные** (`@SpringBootTest @Transactional`, фейковый `GoszakupClient`):
  `TechSpecService` — запись текста+габаритов в лот (PDF-байты собираются PDFBox'ом из
  двуязычного текста с «не более 1200х800х1300 мм, вес не более 45 кг»); идемпотентность
  (повторный вызов не дублирует); рыночный гард (лот RF при KZ-контексте → 404);
  ручной тендер → 400; файла нет → 404 без изменения лота.
- **Гейты:** `./gradlew test` (только 2 известных `ApplyAutoFillServiceTest`), `npm run build`.
- **Живая проверка (Playwright + живой токен):** лот «Пульсоксиметр» (тендер 17280874-1, лот id 3305):
  «ТЗ» → тост, спека в таблице = русская ТТХ («Портативное устройство… Диапазон измерения
  сатурации: 0-100…»). У пульсоксиметра габаритов в ТЗ нет по природе — подбор честно останется
  `noCriteria` (это корректно). Дополнительно поискать в ленте лот габаритной техники
  (УЗИ/рентген/кровать) с «не более …х…х…» в ТЗ — там габариты появятся в колонке и подбор оживёт
  (если такого лота в живой ленте нет — семантика записи габаритов покрыта интеграционным тестом,
  не блокер).

## 7. Вне скоупа (бэклог)

- Авто-разбор при импорте (масс-прогон 1000+ PDF) и/или фоновая очередь.
- LLM-фолбэк для сложных/табличных ТЗ (опт-ин Anthropic API, Haiku).
- Разбор казахской секции (сейчас — русская всегда есть).
- Не-PDF форматы (docx/xlsx техспеки не встречены; при появлении — POI уже в стеке).
- Авто-подбор `equipmentType` лота из текста ТЗ.
- Точный матч goszakup-лота при дублях имён (хранить goszakup lotNumber при импорте).

## 8. Порядок работ (для плана)

1. PDFBox в `build.gradle` + `PdfTextExtractor` (TDD).
2. `TechSpecExtractor` (TDD, реальные фрагменты из разведки).
3. `GoszakupClient.fetchLotTechSpec/downloadFile` (+ фейк для тестов).
4. `TechSpecService` + `TechSpecWriter` + эндпоинт + контроллер-гарды (TDD, фейковый клиент).
5. Фронт: кнопка «ТЗ», тост, дополнение noCriteria-баннера.
6. Полный прогон + живая проверка токеном (Playwright) + CLAUDE.md + тур.

Гочи: миграций НЕТ (пишем в существующие поля); `@FilterDef` не трогаем; правка полей лота через
`save(lot)` безопасна (не orphanRemoval-путь); токен не эхо-печатать; сеть вне транзакции (§6).
