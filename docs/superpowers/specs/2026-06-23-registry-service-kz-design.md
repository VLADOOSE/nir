# Дизайн: Реестр-сервис проверки регистрации медизделий (Казахстан)

**Дата:** 2026-06-23
**Статус:** утверждён, готов к написанию плана
**Модуль:** первый шаг расширения АИС из дипломного проекта в рабочий продукт (тендеры + частники, автоматизация)

---

## 1. Контекст и цель

АИС изначально — учёт участия в тендерах на медоборудование (РФ). Диплом сдан; система превращается
в рабочий инструмент, который дополнительно обслуживает поток **частников** и автоматизирует рутину.
Перенос идей из коммерческой сессии West-Med (KZ) описан в `HANDOFF-АИС-медоборудование.md`.

**Этот модуль** — самый ценный и фундаментальный из переиспользуемых: автоматическая проверка
регистрации медизделия в гос-реестре. Регистрация (РУ) определяет **допуск к поставке** и **НДС-льготу**.

**Целевой рынок v1:** Казахстан (реестр НЦЭЛС). РФ (Росздравнадзор) — в бэклоге, под будущую абстракцию.

**Сценарий v1:** обогащение каталога — к каждой позиции `med_equipment` привязываем № РУ из реестра;
в каталоге появляется статус допуска + НДС-флаг.

**Доменная заметка (от заказчика):** в потоке частников вход — *конкретный аппарат* (производитель +
название модели), а не параметры. Это та же форма, что матчинг против реестра → примитив матчинга
делаем **переиспользуемым** (`(name, manufact) → кандидаты`); поток частников вызовет его позже.

**Бизнес-модель (от заказчика, важно для будущей мульти-рынок задачи):** Регион-Мед (РФ) и West-Med (KZ) —
**две отдельные компании со своими данными** (каталог/тендеры/клиенты раздельны), вместе как один датасет
не работают. Связь — снабженческая: что выгоднее в РФ, закупает Регион-Мед → продаёт и везёт в KZ для
West-Med → West-Med продаёт заказчику (РФ → KZ реэкспорт). Т.е. **Регион-Мед = канал сорсинга/поставщик
для West-Med** (важно для будущего сорсинг-модуля). Реестры разные: KZ — НЦЭЛС, РФ — Росздравнадзор.
В этой итерации работаем только в контексте West-Med (KZ).

## 4. Архитектура и поток данных

Три слоя, чистое разделение ответственности:

```
[офлайн, Node/Puppeteer]  ndda-fetch-all.mjs ──► rk-mi-registry-full.json (дамп 14k РУ)
            │  обход origin-check 405 — только puppeteer из origin страницы
            ▼  импорт (на старте при пустой таблице / по кнопке «Обновить реестр»)
[Postgres]  med_registry  ◄── pg_trgm ──  поиск кандидатов
            │
            ▼  привязка (подтверждает человек)
[каталог]   med_equipment.registration_status + med_registry_id  ──► бейдж допуска + НДС
```

Live-путь (поиск/проверка) — чисто Java/Spring + Postgres, как вся текущая АИС, без второго рантайма
в запросном пути. Puppeteer работает только офлайн (то, что он делает хорошо — обход origin-check).

## 5. Модель данных

**Новая таблица `med_registry`** (живучая — `IF NOT EXISTS`, НЕ в DROP-списке `schema.sql`):
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE TABLE IF NOT EXISTS med_registry (
    id              BIGSERIAL PRIMARY KEY,
    reg_number      VARCHAR(100) NOT NULL UNIQUE,  -- № РУ, естественный ключ (в дампе уникален: 14072/14072)
    name            TEXT NOT NULL,                 -- наименование МИ
    producer        VARCHAR(500),
    country         VARCHAR(200),
    reg_date        DATE,
    expiration_date DATE,
    unlimited       BOOLEAN DEFAULT FALSE,
    imported_at     TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_reg_name_trgm     ON med_registry USING gin (name gin_trgm_ops);
CREATE INDEX IF NOT EXISTS idx_reg_producer_trgm ON med_registry USING gin (producer gin_trgm_ops);
-- UNIQUE на reg_number уже даёт b-tree индекс для lookup/FK
```
Источник полей — slim-схема дампа: `reg`, `name`, `producer`, `country`, `regDate`, `exp`, `unlimited`.

**Новые колонки на `med_equipment`** — связь по `reg_number` (естественному ключу), а **не** по суррогатному `id`,
чтобы реимпорт (`ON CONFLICT (reg_number) DO UPDATE`) не рвал привязки:
```sql
registration_status      VARCHAR(30) NOT NULL DEFAULT 'UNCHECKED',  -- UNCHECKED|REGISTERED|NOT_REGISTERED|NOT_MEDICAL
med_registry_reg_number  VARCHAR(100) REFERENCES med_registry(reg_number),  -- заполняется при REGISTERED
registration_checked_at  TIMESTAMPTZ
```

**НДС-флаг не храним** — вычисляем в response: `vatExempt = (status == REGISTERED)`. Один источник правды.

**Java-enum** `RegistrationStatus { UNCHECKED, REGISTERED, NOT_REGISTERED, NOT_MEDICAL }`.

> ⚠️ В dev `med_equipment` пересоздаётся из `data.sql` на каждом старте → проставленные привязки слетают
> при рестарте (реестр-таблица переживает, связи на оборудовании — нет). Чинится переходом на миграции
> (решение #9 — отложено). v1 спроектирован так, что переход потом дешёвый.

## 6. Алгоритм матчинга

Нативный запрос с `pg_trgm`, на вход — `(name, manufact)`:
```sql
SELECT *,
  (0.6 * similarity(producer, :manufact) + 0.4 * similarity(name, :name)) AS score
FROM med_registry
WHERE producer % :manufact OR name % :name
ORDER BY score DESC
LIMIT :limit;          -- по умолчанию 5
```
Производитель — сильный сигнал (0.6), наименование — поддержка (0.4). Топ-N кандидатов со score →
человек подтверждает один (или помечает `NOT_REGISTERED` / `NOT_MEDICAL`).

> ⚠️ **Ограничение:** триграммы посимвольные → латиница в `manufact` («OMRON») не матчится с кириллицей
> в реестре («ОМРОН»). Частично спасает матч по `name` (бренд в наименовании часто латиницей). Полное
> решение (нормализация/транслитерация) — вне v1; такие случаи закрывает ручной поиск + подтверждение.

## 7. Backend-компоненты и API

**Компоненты** (зеркалят существующий вертикальный срез: entity → repository → service → DTO → mapper → controller):
- `MedRegistry` (entity), `MedRegistryRepository` — нативный trgm-запрос кандидатов, `findByRegNumber`, `count`.
- `RegistryMatchService`:
  - **`findCandidates(name, manufact, limit)`** — переиспользуемый примитив (каталог сейчас, частники потом).
  - `candidatesForEquipment(equipmentId)` — обёртка (берёт `name`+`manufact` из `med_equipment`).
  - `confirm(equipmentId, regNumber)` / `mark(equipmentId, status)` — пишет `registration_status`,
    `med_registry_reg_number`, `registration_checked_at`.
- `RegistryImportService` — читает JSON-дамп (`registry.kz.dump-path`), батч-upsert `INSERT ... ON CONFLICT
  (reg_number) DO UPDATE`; на старте при пустой таблице и по refresh. Идемпотентен, суррогатный `id` стабилен
  → существующие привязки каталога не рвутся.
- Расширение `MedEquipmentResponse`/mapper — встроенный `EquipmentRegistrationResponse`.

**API (`RegistryController`):**
| Метод | Endpoint | Назначение |
|-------|----------|-----------|
| GET | `/api/registry/candidates?name=&manufact=` | общий примитив (ad-hoc, готов для частников) |
| GET | `/api/registry/candidates/equipment/{id}` | кандидаты для позиции каталога |
| GET | `/api/registry/reconciliation?status=` | bulk-список: оборудование + статус + топ-кандидат + счётчик (серверно, без N+1) |
| POST | `/api/equipment/{id}/registration` | `{action: CONFIRM, regNumber}` \| `NOT_REGISTERED` \| `NOT_MEDICAL` \| `RESET` |
| POST | `/api/registry/refresh` | реимпорт из JSON, вернуть число записей |
| GET | `/api/registry/search?q=` | свободный lookup (инлайн-поиск/отладка) |

**DTO:** `RegistryCandidateResponse`, `EquipmentRegistrationResponse` (встраивается в `MedEquipmentResponse`),
`ReconciliationRowResponse`, `RegistrationActionRequest`. **Mapper:** `RegistryMapper`.

## 8. Фронтенд (Angular)

**Страница «Сверка с реестром»** (`/registry-reconciliation`):
- Таблица каталога, фильтр по статусу (по умолчанию — непроверенные).
- Строка: название + производитель + бейдж статуса + топ-кандидат (держатель/score).
- Разворот → 5 кандидатов со score-барами + действия: ✓ Подтвердить / выбрать другого / Нет в реестре /
  Не медизделие / Сбросить.
- Поле свободного поиска в строке — фолбэк для трудных случаев (кросс-скрипт).
- В шапке — «Обновить реестр» (показывает число загруженных записей).

**Инлайн в карточке оборудования** (read-only):
- Блок «Регистрация РК»: бейдж статуса + НДС-бейдж (Льгота / Облагается 12%) + реквизиты РУ
  (№, держатель, страна, срок/бессрочно).
- «Изменить привязку» → ведёт на bulk-экран с фокусом (`?focus=id`). Без дублирующего UI редактирования.

## 10. Обработка ошибок

- Импорт: дамп отсутствует/битый → лог + не трогаем существующие строки, вернуть ошибку + счётчик.
- Матч: пустой `manufact` → матч только по `name`; нет кандидатов → пустой список, доступны `NOT_REGISTERED`/`NOT_MEDICAL`.
- Confirm: неизвестный `regNumber`/`equipmentId` → 400/404. Конкурентные правки → last-write-wins.
- `pg_trgm` отсутствует → внятная ошибка на старте.

## 11. Тестирование

- **Unit:** ранжирование `RegistryMatchService` (сид-строки → ожидаемый порядок); идемпотентность импорта
  (парсинг sample-JSON → строки, повторный прогон не дублирует); переходы статусов.
- **Интеграция:** нативный trgm-запрос возвращает ранжированных кандидатов — нужен `pg_trgm` → Testcontainers
  Postgres (на этапе плана проверить наличие тест-инфры в проекте; добавить при отсутствии).
- **E2E-смоук:** старт → реестр импортирован (≈14072) → «Сверка» → известная зарегистрированная позиция
  показывает кандидата → подтверждение → карточка показывает `REGISTERED` + НДС-льгота.
