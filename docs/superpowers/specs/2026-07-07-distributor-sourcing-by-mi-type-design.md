# Дизайн: подбор поставщиков по виду МИ (отправка КП, подблок 1)

- **Дата:** 2026-07-07
- **Статус:** согласовано (brainstorming), готов к writing-plans
- **Рынок:** прежде всего KZ (West-Med), механика рынко-нейтральна
- **Тема-родитель:** «грамотно отправлять КП и получать ответы» (3 подблока, ниже)

## 1. Контекст и проблема

Поток West-Med: клиника называет бренд/модель → проверяем реестр НЦЭЛС → запрашиваем КП у
дистрибьюторов. Но на **госзакуп-KZ тендерах** лоты приходят **без бренда и без типа**
(`tender_lot.equipmentType` пуст), а панель подбора поставщиков (`LotSourcingService.build`,
`GET /api/tenders/{id}/lot-sourcing`) матчит **только по бренду** и показывает **ВСЕХ**
дистрибьюторов рынка подряд. Реальных KZ-дистрибьюторов в базе нет (2 фейковых из V2-сида).

Итог: оператор не видит, кому осмысленно слать КП под данный вид МИ. Нужно, чтобы под лот
подтягивались **правильные** поставщики по **виду МИ** (специализации), а не весь список.

## 2. Границы

Родительский блок распадается на 3 зависимых подблока (делаем последовательно, каждый — свой
спек/план):

1. **[ЭТОТ СПЕК] Данные + подбор по виду МИ** — наполнить базу реальными KZ-дистрибьюторами с
   отметками (бренды + виды МИ) и сделать так, чтобы под лот подтягивались правильные поставщики.
2. Грамотная отправка КП (выбор из подобранных, тело письма, статусы SENT/RESPONDED в UI, повтор).
3. Приём и структурный разбор ответов поставщика (цена/срок/модель → `PriceRequestItem`).

Подблоки 2–3 — **вне этого спека** (перечислены в §13 как follow-up).

### Что НЕ трогаем (уже готово)
- Модель `Distributor.equipmentTypes` (M2M на `equipment_type`) + `Distributor.brands`
  (`@ElementCollection`).
- Редактор в карточке дистрибьютора (`distributors.component.ts`): чипы брендов +
  чекбоксы типов; `DistributorRequest.equipmentTypeIds/brands`; показ «Все типы» при пустых типах.
- `BrandMatch`, реестр-кандидаты по лоту (`RegistryMatchService.candidatesForLot`),
  `PriceRequestSendService` (единый канал отправки КП).

## 4. Таксономия «видов МИ»

Остаётся справочник `equipment_type` (общий, без рынка). Сейчас 10 диагностика-центричных типов
(УЗИ, Рентген, ИВЛ, Монитор пациента, КТ, МРТ, ЭКГ, Дефибриллятор, Лабораторный анализатор,
Эндоскоп). На этапе данных (V7) **расширяю справочник** под реальные KZ-закупки, опираясь на
портфели найденных дистрибьюторов и реальные имена goszakup-лотов. Кандидаты на добавление
(финализирую по факту ресёрча): Хирургия/эндохирургия, Стерилизация/дезинфекция,
Реанимация/анестезия (наркозно-дыхательная), Физиотерапия, Стоматология, Лучевая диагностика
(маммограф/ангиограф/С-дуга), Расходные материалы/реагенты, Мед. мебель. Тот же справочник —
«виды МИ» для отметок дистрибьютора (двойного смысла не заводим).

**Расширяем умеренно — до настоящих категорий аппаратов, не до мелочи.** Дробить справочник до
«электродов»/«силиконовых прокладок» бессмысленно (бесконечная гранулярность). Мелочь/аксессуары
ловит не тип, а **точечный поиск по бренду/аппарату (§8.2)** — дискриминатор там аппарат, к которому
принадлежность («электроды для Элэскулап» → тот, кто возит Элэскулап), а не вид МИ.

## 5. Изменения модели данных

### 5.1 Новый обучаемый словарь `equipment_type_synonym` (миграция V7)
По образцу `header_synonym`:
```
CREATE TABLE equipment_type_synonym (
  id                BIGSERIAL PRIMARY KEY,
  term_norm         VARCHAR(255) NOT NULL UNIQUE,   -- нормализованный термин (lower, trim)
  equipment_type_id BIGINT NOT NULL REFERENCES equipment_type(id)
);
```
- Общий (без рынка), как `equipment_type`.
- Сид очевидными синонимами (узи/ультразвук/сканер→УЗИ; ивл/вентилятор лёгких/наркозно-дыхательный
  →ИВЛ; флюорограф/рентгеновск*→Рентген; электрокардиограф/экг→ЭКГ; гематологический/биохимический/
  анализатор→Лаб.анализатор; гастроскоп/колоноскоп/эндоскоп→Эндоскоп; дефибриллятор→Дефибриллятор;
  монитор пациента/прикроватный монитор→Монитор; компьютерн* томограф→КТ; магнитно-резонанс*→МРТ;
  …расширю вместе с новыми типами).
- **Обучаемость:** при ручной смене вида МИ на лоте (§7) добавляем `term_norm` головного токена
  лота → выбранный тип, если такого термина ещё нет (best-effort, не критично).

### 5.2 `tender_lot.equipmentType` — уже существует
Колонка `equip_type_id` есть (`TenderLot.equipmentType`), просто не заполняется на импорте и не
сохраняется формой лота (дефект). Персист вида МИ (§7) её наполняет. Новой миграции не требует.

### 5.3 Сид реальных KZ-дистрибьюторов (миграция V7)
`INSERT` в `distributor` (market=KZ) + `distributor_brand` + `distributor_equipment_type` по
результатам веб-ресёрча. Поля: name, city/address, website, контакт (если публичен), phone;
**email оставляю NULL или с пометкой** — пользователь верифицирует перед боевой рассылкой.
Фейковые «ТОО МедСнаб/Алматы Медтехника» из V2 — заменяю/дополняю (V2 не правим, добавляем в V7;
при желании чистку фейка тоже в V7 `DELETE ... WHERE name IN (...)`).

## 6. Классификатор лот → вид МИ (`LotTypeClassifier`)

Новый сервис `service/LotTypeClassifier` (+ репозиторий/загрузка словаря
`EquipmentTypeSynonymRepository`).

- **Вход:** `TenderLot` (используем `equipName` + `requiredSpec`).
- **Нормализация/токенизация:** переиспользуем стоп-слова канцелярита из `LotQueryTokenizer`
  (номера/места/сроки поставки — мусор). Головной токен (первое значимое слово имени) весит больше.
- **Матч:** по словарю `equipment_type_synonym` (нормализованный `contains`/по-токенно). Несколько
  типов совпало → берём по покрытию/весу токенов.
- **Выход:** `record TypeGuess(Long typeId, String typeName, double confidence, List<TypeGuess> alternatives)`.
  `confidence` ∈ [0,1] — доля значимых токенов, попавших в тип / позиция головного токена.
- Пустой результат допустим (unknown) → деградация в §8.
- **Голден-тест** на реальных именах лотов.

## 7. Персист вида МИ на лоте

Новый эндпоинт `POST /api/lots/{id}/equipment-type` body `{ "typeId": <id|null> }`:
- Пишет `lot.setEquipmentType(...)` (typeId=null → снять).
- **Гард рынка + сеть-вне-tx паттерн (§6 CLAUDE.md):** лот через `em.find` минует hibernate-фильтр →
  явная проверка `lot.tender.market == MarketContext.get()`; запись в отдельном `@Transactional`-бине.
- Обучение словаря (best-effort): добавить головной токен имени лота → typeId в
  `equipment_type_synonym`, если термина нет.
- Заодно **чиним дефект** сохранения типа: форма лота (`TenderLotRequest.equipTypeId`) и/или эта
  ручка — единый путь записи типа. (Точную точку интеграции формы уточню в плане; минимально —
  ручка + селектор в панели КП.)
- `@PreAuthorize("hasRole('ADMIN')")` как прочие записи.

## 8. Подбор поставщиков (ядро — расширяем `LotSourcingService.build`)

Два уровня в одном ранге: **Tier 1** — ранг по виду МИ + бренду (§8.1); **Tier 2** — точечный
поиск по бренду/аппарату для мелочи/аксессуаров (§8.2). Tier 2 не отдельная выдача, а
дополнительный **источник brand-сигнала** + поле ввода в UI.

### 8.1 Ранжирование по виду МИ + бренду (Tier 1)

Сейчас метод собирает `BrandSource` (бренд предложенной модели или производители реестр-кандидатов)
и по каждому дистрибьютору считает brand-hits, `preselect = !hits.isEmpty()`, возвращает ВСЕХ.

**Новое:** к brand-сигналу добавляем type-сигнал и ранжируем.

Для лота(ов) определяем набор видов МИ:
- одно-лотовый режим: `detectedType` из `lot.equipmentType` (если уже сохранён) иначе
  `LotTypeClassifier` (подсказка);
- мульти-лотовый режим (несколько выбранных лотов): **объединение** типов и брендов всех лотов.

Для каждого дистрибьютора рынка (`distributorService.findAll()`):
- `typeHit` = `distributor.equipmentTypes` содержит вид МИ лота. «Универсал» (пустой
  `equipmentTypes`) — **не** typeHit, но и не исключаем: идёт ниже типизированных, с меткой
  «все виды».
- `brandHit` = существующий `BrandMatch.firstCarried(...)` по набору `BrandSource`. Набор =
  { бренд предложенной модели, производители реестр-кандидатов (как сейчас), **эффективный термин
  Tier 2 (§8.2)** }.
- `score = 1.0·brandHit + 0.7·typeHit (+0.3 если оба)`.
- `relevant = typeHit ∨ brandHit`.

Сортировка: `relevant` по `score` desc (универсалы — в хвосте релевантных при равенстве); затем
нерелевантные (нет сигналов) — сворачиваются под «ещё N». `preselect` = сильнейшие (brandHit или оба).

**Причины (reasons):** список меток для UI — `["ИВЛ"(type), "Hamilton"(brand)]`.

### 8.2 Точечный поиск по бренду/аппарату (Tier 2)

Для мелких/аксессуарных лотов вид МИ — грубый инструмент; дискриминатор — **бренд/аппарат**.

- **Эффективный термин поиска** выводим по приоритету:
  1. бренд предложенной модели (`lot.proposedEquipment.manufact`), если принята;
  2. иначе термин аппарата из **`ComplectTermExtractor`** (аксессуарный лот — переиспользуем прошлую
     фичу комплектности: «электроды для Элэскулап» → «Элэскулап»);
  3. иначе производитель топ реестр-кандидата / головной токен имени лота.
- Термин **редактируем оператором** (штурвал при промахе эвристики) и передаётся в запрос
  `lot-sourcing` (см. §9).
- Матч термина против `distributor.brands` (`BrandMatch` substring сейчас; фаззи/транслит —
  follow-up) → даёт `brandHit` с `reason.kind=BRAND, label=<совпавший бренд>`. То есть Tier 2 — это
  **источник brand-сигнала**, встроенный в тот же ранг (§8.1), плюс явное поле ввода в UI.
- Результат: «электроды для Элэскулап» **без всякого типа «электроды»** находят дистрибьютора,
  возящего Элэскулап. Так мелочь решается без раздувания таксономии.
- Только одно-лотовый режим (у аксессуара свой аппарат); в мульти-лоте поле не показываем.

### Деградация / edge cases
- Вид МИ не определён (unknown) и брендов нет → фолбэк на текущий brand-only ранг + баннер
  «укажите вид МИ, чтобы подобрать по специализации» (по образцу `noCriteria`/комплектности).
- Ни одного дистрибьютора на рынке → пустое состояние с подсказкой «наполните справочник».
- Мульти-лот: инлайн-селектор типа **не** показываем (типы разных лотов) — только объединённый
  ранг; смена типа — в одно-лотовом режиме или в карточке лота.

## 9. API

- **`GET /api/tenders/{id}/lot-sourcing`** (расширяем `LotSourcingResponse`):
  - опц. query `term` (Tier 2, §8.2) — переопределяет эффективный термин поиска поставщика;
    при отсутствии сервис выводит термин сам (приоритет из §8.2).
  - top-level: `detectedType { id, name, confidence } | null`, `typeAlternatives: [{id,name}]`
    (только одно-лотовый режим), `singleLot: boolean`, `sourcingTerm: string | null`
    (эффективный термин Tier 2 — для префилла редактируемого поля на фронте).
  - `Entry`: добавить `relevant: boolean`, `score: double`, `reasons: [{ kind: "TYPE"|"BRAND",
    label: string }]`. Существующий `matchedBrands` **оставляем** (обратная совместимость);
    `reasons` — расширенный суперсет (типы + бренды), фронт рисует его.
- **`POST /api/lots/{id}/equipment-type`** `{ typeId }` → `200` (обновлённый лот/тип) — §7.
- Справочник типов для селектора — существующий `GET /api/equipment-types`.

## 10. Фронт (`tenders.component.ts`, панель КП/подбора)

- В шапке панели КП (одно-лотовый режим): селектор **«Вид МИ»** (`<select>` из
  `/api/equipment-types`), преселект = `detectedType`, чип уверенности; `change` →
  `POST /api/lots/{id}/equipment-type` → перезапрос `lot-sourcing`.
- Рядом — редактируемое поле **«Поиск поставщика»** (Tier 2, §8.2), префилл = `sourcingTerm`,
  штурвал при промахе; `change` → перезапрос `lot-sourcing?term=<...>` (тот же паттерн
  редактируемого термина, что в поиске по комплектности).
- Список поставщиков: блок **«Релевантные»** (сортировка по score, ★ и зелёная подсветка лучшего)
  + сворачиваемый **«ещё N нерелевантных»** — переиспользую стиль комплектности
  (`_relevant`/`_zero`/`_showZero` паттерн).
- Причины — маленькие чипы `✓ИВЛ` (type) / `✓Hamilton` (brand).
- Баннер деградации при `detectedType==null && нет брендов`.
- `cdr.detectChanges()` после async (паттерн проекта).
