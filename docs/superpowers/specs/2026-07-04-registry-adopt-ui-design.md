# «Взять из реестра в работу» + чистка UI карточки тендера — дизайн

**Дата:** 2026-07-04. **Ветка:** `feat/registry-adopt-ui`. **Статус:** одобрен («ок»).

## 1. Зачем

Сквозной конвейер KZ рвётся на шаге «выбрать модель»: «Подобрать» смотрит в пустой KZ-каталог,
а «Реестр» находит зарегистрированные модели (REGIUS/VistaScan), но запись реестра — тупик
(не каталог: без цены/поставщика, нельзя утвердить моделью лота). Мост: реестр-кандидат →
позиция каталога → предложенная модель лота → КП. Оператор подтверждает кандидата вручную.
Заодно чистим карточку тендера от мёртвых элементов (у 394/394 импортных пусты способ
закупки/контакты/facility; у 1027/1027 лотов пуст тип; колонка «Тип» вообще рисует
несуществующее поле `l.equipType` и пуста везде).

## 2. Часть 1 — мост «Взять в работу»

### Бэкенд
- `RegistryMatchService.adoptForLot(Long lotId, String regNumber): TenderLot` (`@Transactional`;
  сервис уже держит реестр+каталог+лоты):
  1. лот + гард рынка через тендер (`lot.getTender().getMarket() != MarketContext.get()` → 404;
     `findById` = `em.find` обходит фильтр — паттерн proposed-equipment);
  2. `MedRegistry reg = registryRepository.findByRegNumber(regNumber)` → 404 «РУ не найдено»;
  3. дедуп: `medEquipmentRepository.findFirstByRegistrationRegNumber(regNumber)` (derived finder →
     аспект включает рыночный фильтр под привязанной сессией → ищет только на текущем рынке);
     найдено → переиспользуем;
  4. иначе создаём `MedEquipment`: `name = trim255(reg.name)`, `manufact = trim255(reg.producer)`
     или «не указан» при null (колонка NOT NULL), `equipmentType/габариты = null`,
     `registrationStatus = REGISTERED`, `registration = reg`, `registrationCheckedAt = now`;
     market штампуется листенером/пред-штампом;
  5. `lot.setProposedEquipment(eq)` + save (правка поля — безопасно, не orphanRemoval-путь).
     Повторный adopt другого РУ заменяет модель.
- Репозиторий: `Optional<MedEquipment> findFirstByRegistrationRegNumber(String regNumber)`.
- Эндпоинт: `POST /api/lots/{id}/adopt-registry` (ADMIN, `TenderLotController` — RegistryMatchService
  уже заинжекчен) с телом `{regNumber}` (`AdoptRegistryRequest`, `@NotBlank`) → `TenderLotResponse`.

### Фронт (`tenders.component.ts`)
- В строке панели «Реестр» кнопка **«Взять в работу»** → `POST adopt-registry` → тост
  «Модель из реестра предложена для лота» → закрыть панель реестра → `loadLots()` →
  **сразу `openKpPanelFor(lot)`** (КП-панель по этому лоту; предотметка поставщиков по бренду
  производителя сработает сама — lot-sourcing/PROPOSED_MODEL).
- `ApiService.adoptRegistryForLot(lotId, regNumber)`.

## 3. Часть 2 — чистка UI (правило «пусто — не рисуем», на обоих рынках)

- **Инфо-сетка карточки:** «Способ закупки», «Контактное лицо», «Телефон», «Эл. почта»,
  «Адрес поставки», «Окончание приёма заявок», «Дата публикации» — `*ngIf` по значению.
  «Заказчик», «Статус», «Начальная цена» — всегда.
- **Таблица лотов:** колонки «Тип»/«Габариты (макс.)»/«Макс. вес» показываются только если хоть
  у одного лота заполнены (`hasAnyType()/hasAnyDims()/hasAnyWeight()` по загруженным lots;
  ячейки под теми же условиями). Биндинг типа чинится: `l.equipmentType?.name` (было мёртвое
  `l.equipType`). После разбора ТЗ с габаритами колонка сама возвращается.
- **Кнопки:**
  - «Подобрать» — только `lotHasCriteria(l)`: есть `equipmentType` или любой `max*` (иначе
    smart-match честно бесполезен);
  - bulk «Запросить КП по всему тендеру» — только `!isImportedTender()`;
  - «Реестр» — только `isKz()` (на РФ реестр НЦЭЛС нерелевантен; RF-реестр — бэклог);
  - «ТЗ» (уже только импортные KZ), «КП», «Редактировать», «Удалить» — без изменений.
- Форму лота не трогаем (место ручного ввода). Хардкод-селект типов в форме (УЗИ/Рентген/ИВЛ/
  Монитор строкой, которая никуда не сохраняется — `TenderLotRequest` ждёт `equipTypeId`) —
  в бэклог как отдельный дефект.

## 4. Тесты и live

- **Интеграционные** (`RegistryAdoptTest`): adopt создаёт позицию каталога (REGISTERED, привязка РУ,
  market текущий) и ставит модель лота; дедуп по РУ (второй adopt того же РУ на другом лоте не
  создаёт дубль); замена модели при adopt другого РУ; лот чужого рынка → 404; РУ не существует →
  404; producer=null → manufact «не указан»; name >255 → обрезан.
- **Гейты:** `./gradlew test` (только 2 известных), `npm run build`.
- **Live (Playwright):** тендер 852 (KZ) → «Реестр» → «Взять в работу» на REGIUS →
  бейдж «Предложено: … РУ ✓» в таблице + КП-панель открыта; карточка тендера без «—»-полей;
  колонки Тип/Габариты исчезли (у 852 лот без типа/габаритов); «Подобрать» скрыта на лоте без
  критериев; RF-тендер (openId=1) — поля/колонки на месте (данные есть), «Реестр» скрыт,
  bulk-кнопка на месте.

## 5. Вне скоупа

- LLM-подтверждение «подходит ли РУ под ТЗ» (опт-ин, поверх).
- Хардкод-селект типов в форме лота → справочник equipment-types (бэклог-дефект).
- RF-реестр Росздравнадзора.
- Автосоздание позиций каталога при импорте (каталог наполняется по ходу работы оператора).

## 6. Порядок работ

1. Бэкенд: finder + `adoptForLot` + эндпоинт + `RegistryAdoptTest` (TDD).
2. Фронт: кнопка «Взять в работу» + флоу до КП-панели; чистка UI (инфо-поля, колонки, кнопки).
3. Гейт + live + CLAUDE.md + ревью + мерж.

Гочи: миграций нет; `@FilterDef` не трогаем; adopt пишет через save(lot)/save(eq) — не
orphanRemoval; дедуп-finder обязан идти под `@Transactional` (рыночный фильтр).
