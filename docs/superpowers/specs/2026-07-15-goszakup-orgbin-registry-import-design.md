# Реестр-ориентированный импорт тендеров goszakup по `orgBin`

**Дата:** 2026-07-15
**Статус:** дизайн утверждён, ждёт плана реализации
**Рынок:** KZ (goszakup.gov.kz). SK-Фармация и РФ не затрагиваются.

## 1. Проблема (с доказательствами)

Оператор заметил: по ЗКО (Западно-Казахстанская область) в системе мало тендеров, хотя заказчиков-больниц там десятки. Расследование против живого goszakup API (токен «Унифицированных сервисов»):

- **По 29 больницам ЗКО за последние 30 дней goszakup отдаёт 471 тендер. Текущий импорт пропускает только 14 (3%). 457 (97%) отсекаются ступенью-1 (keyword-предфильтр по ИМЕНИ объявления) — до того, как мы вообще смотрим лоты.**
- Большая часть из 457 — законно не-медтовары (ремонт кондиционеров, овощи, дрожжи, обучение, страхование). Их отсекать правильно.
- Но keyword-фильтр убивает и **реальную медтехнику**, потому что матчит фиксированные фразы по имени:
  - «Государственный закупки **медицинских изделий**» (×6, одно ACTIVE) — `изделия медицинск` ≠ `медицинских изделий` (обратный порядок слов).
  - «Приобретение **облучателя**» — keyword `облучатель`, в имени `облучателя` (родительный падеж).
  - «Приобретение **медицинских весов**», «**холодильник медицинский**» — терминов нет в списке.
  - «Приобретение **товара для медицинской организации**» — дженерик-имя, техника видна только в лотах.

Три структурных факта, проверенных по API:

1. **`customerBin` — НЕ фильтруемое поле** в v3 GraphQL (`Unknown field`). Веб-поиск оператора (`filter[customer]`) работает на другом бэкенде — скопировать этот параметр в API нельзя.
2. **`orgBin` (и `orgNameRu`) — фильтруемые** и не используются. `TrdBuy(filter:{orgBin:"<БИН>"})` возвращает ровно тендеры одной больницы (снял по 200+ на каждую). Это API-нативный способ сделать то, что оператор делает руками.
3. Проверка «медтовар или нет» **по лотам** надёжна: лоты называют конкретику. Живой прогон классификатора: «Облучатель» → KEEP; «Йодид калия» (лекарство) / «Мыло» / «Помидор» / «Сыворотка» → DROP — корректно.

**Вывод:** импорт теряет тендеры не потому, что ЗКО пустой, а потому что душит их на lossy-фильтре по имени. Чиним: (а) таргетим известные больницы по `orgBin`, (б) «медицинскость» решаем по лотам, не по имени.

## 2. Цели / не-цели

**Цели:**
- Импорт goszakup тянет тендеры из реестра медорганизаций по `orgBin`, а не из региональной KATO-ленты.
- Классификация медтовара — по лотам (оборудование + расходка НЦЭЛС), не по имени. Лекарства/еда/хозтовары/услуги отсекаются.
- Реестр больниц живёт в существующей сущности `facility` и управляется на странице `/facilities` (не новая страница/сущность).
- Сид — 29 больниц ЗКО. Другой регион = добавить записи `facility` (миграцией или через UI).
- Регион импортированного тендера берётся из реестра больницы (точно), а не угадывается `RegionResolver`.

**Не-цели (YAGNI):**
- Сбор БИНов других регионов сейчас (только ЗКО; расширяемо).
- НЦЭЛС-матч как классификатор лотов (оставлен как будущий апгрейд точности — инфра `RegistryMatchService` есть).
- Связывание импортированных тендеров с `facility` (остаётся денормализованный `customer_bin`/`customer_name`, `facility=null` как сейчас).
- Гибридный «страховочный» проход по региональной ленте для заказчиков не из реестра.
- БИНы централизованных организаторов (УЗ акимата, закупающих ЗА больницы) — при необходимости добавляются как обычные записи реестра позже.

## 3. Изменения данных

### 3.1 `facility` (Flyway V13)
Добавить две колонки:
- `region VARCHAR(100)` — канонический регион (значения = `REGIONS` фронта / `RegionResolver`), напр. `Западно-Казахстанская область`. Nullable.
- `monitor_tenders BOOLEAN NOT NULL DEFAULT false` — «тянуть ленту тендеров этой организации по `orgBin`».

**БИН** переиспользует существующее поле `inn` (`VARCHAR(12)` — казахстанский БИН ровно 12 цифр, валидатор `FacilityRequest` уже принимает `^(\d{10}|\d{12})$`). Отдельная колонка не заводится.

Сущность `Facility`: `+ String region;` `+ Boolean monitorTenders;`. Остаётся market-scoped (`MarketScoped`, `@Filter marketFilter`, `MarketStampingListener`) — **`@FilterDef` не трогаем** (он только на `Tender`, §6/§14 CLAUDE.md).

### 3.2 Сид 29 больниц ЗКО (в V13)
Идемпотентные INSERT (guard `WHERE NOT EXISTS (SELECT 1 FROM facility WHERE inn = '<БИН>' AND market='KZ')`): `name` (офиц. наименование), `inn`=БИН, `region`='Западно-Казахстанская область', `market`='KZ', `monitor_tenders`=true; `address`/`phone`/`email` где оператор дал (см. Приложение A). Контактное лицо (last/first/middle) не заполняем (даны ФИО руководителя, не отдела закупа). `facility.name` — UNIQUE: офиц. наименования уникальны и не конфликтуют с демо-сидом. SQL-экранирование кавычек-ёлочек и казахских букв — на реализации.

## 4. Компоненты (границы и интерфейсы)

### 4.1 `GoszakupClient` / `GoszakupHttpClient` — новый метод
**Что:** `TrdBuyV3PageDto fetchTrdBuyPageByOrgBin(String orgBin, Long after)`.
**Как:** тот же v3-GraphQL, что `fetchTrdBuyPageByKato`, но `filter:{orgBin:$o}` (`$o:String`). Пагинация `after`/id DESC, тот же набор полей и алиасы (`number_anno:numberAnno` …), тот же парс `extensions.pageInfo`.
**Рефактор:** общий постинг v3 вынести в приватный `postTrdBuyV3(query, vars) -> TrdBuyV3PageDto`; `...ByKato` и `...ByOrgBin` строят свой query+vars и зовут его. Не дублировать разбор ответа.
**Зависимости:** ObjectMapper, HTTP, Bearer-токен (как есть).

### 4.2 `FacilityRepository` — методы выборки реестра
- `List<Facility> findByMarketAndMonitorTendersTrue(Market market)`
- `List<Facility> findByMarketAndRegionAndMonitorTendersTrue(Market market, String region)`

Рынок передаём **явно** (`Market.KZ`) — не полагаемся на hibernate-фильтр (фоновый поток импорта может не иметь привязанной сессии, §6). Читаем только скаляры (`inn`/`region`/`name`) — LAZY-коллекций нет, OSIV не нужен.

### 4.3 `GoszakupImportService` — путь `importByRegistry`
**Что:** заменяет KATO+keyword путь.
```
importByRegistry(region, sum):
  orgs = region==null ? repo.findByMarketAndMonitorTendersTrue(KZ)
                      : repo.findByMarketAndRegionAndMonitorTendersTrue(KZ, region)
  sum.orgsTotal = orgs.size
  cutoff = today - sinceDays (30, конфиг)
  для org в orgs (у кого inn не пуст):
     sum.currentOrgName = org.name
     страницами fetchTrdBuyPageByOrgBin(org.inn, after):
        для тендера d:
           fetched++
           publishDate < cutoff → skipped++, continue
           importOne(d, sum, regionOverride = org.region)   // лоты + MedicalRelevanceFilter + upsert
        стоп-пагинация, когда вся страница старше cutoff, иначе after=nextAfter
     sum.orgsProcessed++
```
**Ключевые отличия от старого пути:** нет `keywordOk`; регион задаётся из реестра (`regionOverride = org.region`), а не резолвится. `importOne` переиспользуется как есть (фетч лотов → `MedicalRelevanceFilter.isRelevant` → `writer.upsertOne(d, subj, lots, regionOverride)`); упрощается учёт счётчиков (см. 4.6). Ошибка одной больницы изолирована (try/catch вокруг цикла по org: `errors++`, продолжаем).

**Ретайр старого:** `fetchWholeFeed`, `fetchRegionFeed`, `keywordOk` и использование `KatoDictionary` из сервиса удаляются; точки входа (ручной запуск + автополлинг) зовут `importByRegistry`. `KatoDictionary` и `fetchTrdBuyPageByKato` в клиенте можно оставить (не мешают) либо удалить при чистке — решается в плане по факту зависимостей тестов.

### 4.4 `MedicalRelevanceFilter` — дочистка по живым данным
Классификатор по лотам (POSITIVE есть И NEGATIVE нет). Правки:
- **POSITIVE — стеммить окончания** (родит./мн.ч.): `облучатель`→`облучател`, и т.п., чтобы падежные формы в именах лотов матчились. Добавить наблюдённые термины оборудования/расходки: `весы медицин`, `холодильник медицин`, `концентратор кислород`/`кислородн концентратор`, `кроват функционал`, `рециркулятор`, `бактерицидн`, `облучател`, `ингалятор`/`небулайзер`, `пульсоксиметр`, `негатоскоп`, `дозатор`, `кушетк`, `каталк`/`тележк медицин`.
- **NEGATIVE — добавить** сервис-сигналы: `поверк`, `метролог` (поверка/метрология СИ — услуга), сохранить существующие.
- Лекарства (INN-названия: `Йодид калия`, `Сульфадиазин`, `Глюконат кальция`) POSITIVE не содержат → отсекаются сами; спец-негатив по лекарствам не нужен.
- Правило «тендер релевантен при ≥1 медтоварном лоте» — без изменений.

Границы: чистая функция от текста лота, без сети/БД. Golden-тест фиксирует реальные кейсы.

### 4.5 `GoszakupTenderWriter` — без изменений контракта
`upsertOne(d, subj, lots, regionOverride)` уже перезаписывает регион, когда `regionOverride != null`. Передаём `org.region`. `facility` остаётся null, customer — денормализованный (`customer_bin`/`customer_name`). Гард рынка (аспект + `MarketContext.set(KZ)` в фоне) как есть.

### 4.6 `ImportSummary` / прогресс
Добавить: `orgsTotal`, `orgsProcessed`, `currentOrgName`. Счётчики пути orgBin: `fetched` (тендеров просмотрено), `created`, `updated`, `skipped` (не медтовар / старше cutoff), `errors`. «Подходящих» в UI = `created+updated`. Стадийный `matched` из keyword-эпохи в этом пути не используется (учёт `importOne` упрощается: на не-медтовар — `skipped++`, без −1 к matched).

### 4.7 `GoszakupImportScheduler` / `TenderController` — точки входа
Сигнатуры не меняются: `POST /api/tenders/import-kz?region=` → `scheduler.startAsync(region)` → `importByRegistry(region, sum)` в фоне (single-thread executor, `MarketContext.set(KZ)`, §6). `GET /import-kz/status` возвращает тот же `ImportStatus` (+новые поля summary). Автополлинг (`GOSZAKUP_IMPORT_ENABLED`) тоже идёт через `importByRegistry(null)`.

### 4.8 Фронт `/facilities` (`facilities.component.ts`)
Только при `isKz()`:
- Форма: выпадающий **Регион** (`REGIONS` — тот же список, что в `tenders.component`; вынести в общий const, чтобы не плодить третью копию), чекбокс **«Мониторить тендеры (goszakup)»**; ярлык поля `inn` → **«БИН»** (на RF остаётся «ИНН»).
- Список: колонка **Регион** и бейдж **«тендеры»** у `monitorTenders=true`.
- `FormGroup` +`region`,`monitorTenders`; payload create/update включает их. Для KZ+monitor валидируем непустой БИН (клиентски + мягко на бэке).
- Кнопка «Обновить тендеры — <регион>» на `/tenders` не меняется; прогресс читает `orgsProcessed/orgsTotal/currentOrgName` и рисует «Больница N/M · …».

DTO: `FacilityRequest`/`FacilityResponse` +`region`(String)`,`monitorTenders`(Boolean); MapStruct мапит по имени (updateEntity уже `IGNORE` для null — частичные апдейты ок).

## 5. Обработка ошибок
- Нет токена goszakup / сеть → как сейчас (сервис не падает; `errors`/message в summary).
- Ошибка API по одной больнице → `errors++`, следующая больница.
- Пустой реестр для scope → `sum.message = "нет медорганизаций (monitor) в реестре для региона …"`, `enabled=false`.
- Лоты недоступны (404/сеть) → `MedicalRelevanceFilter` фолбэк по имени объявления (как сейчас); при полном провале лота — `errors++`, тендер пропускается.
- Дубль-запуск во время прогона → текущий статус (как сейчас).

## 6. Тестирование
- **Юнит `MedicalRelevanceFilter`** (golden, реальные лоты): Облучатель/Облучателя, Весы медицинские, Холодильник медицинский, Концентратор кислорода → KEEP; Йодид калия, Сульфадиазин, Мыло, Помидор, Бумага, Стартер, «Услуги по обучению», «Поверка СИ» → DROP.
- **Интеграционный импорт** (мок `GoszakupClient`): `fetchTrdBuyPageByOrgBin` отдаёт канонные страницы + лоты; реестр = пара `facility` KZ monitor=true (разные регионы); проверяем: в БД только медтоварные тендеры, `region` = регион больницы, чужой рынок не течёт, `orgsProcessed==orgsTotal`.
- **Репозиторий** `FacilityRepository` (findByMarketAndMonitorTendersTrue[+Region]).
- **Миграция**: V13 накатывается на nirdb, 29 больниц видны под рынком KZ, `monitor_tenders=true`.
- **Аудит существующих тестов goszakup-импорта**: тесты, завязанные на `fetchWholeFeed`/`fetchRegionFeed`/keyword, мигрировать на orgBin-путь или заменить. `GoszakupDtoJsonTest`/`GoszakupHttpClientTest` (живые формы) сохранить; добавить кейс orgBin-запроса.
- **Фронт** — гейт `cd frontend && npm run build`.
- **Живой прогон** (Playwright, рынок KZ, ЗКО): «Обновить тендеры — ЗКО» → в списке появляются медтендеры больниц (Облучатель/весы/холодильник и пр.), которых раньше не было; проверить «куда смотреть».

Гейт зелёного: `./gradlew test` — 0 падений (sandbox off, §5/§13).
