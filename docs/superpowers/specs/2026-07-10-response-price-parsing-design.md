# Дизайн: эвристический парс цены из ответа поставщика (ч.3b, MVP)

**Дата:** 2026-07-10
**Ветка:** `feat/response-price-parsing`
**Статус:** одобрено, реализация

## Контекст и проблема

Поток «отправка КП» (§9 CLAUDE.md) замкнут end-to-end на боевой почте: КП уходит поставщику (SMTP),
поставщик отвечает Reply → `MailReceiveService.poll()` ловит письмо по токену `[КП-id]`, ставит
`PriceRequest.status = RESPONDED`, кладёт текст в `PriceRequest.note`, пишет `InboundEmail`.

**Пробел:** цена из ответа не извлекается — `PriceRequestItem.responsePrice` остаётся `null`, оператор
вписывает вручную в «Калькулятор наценки». Это ч.3b роадмапа.

**Решение (выбрано оператором):** эвристический парсер (regex), без LLM/внешних зависимостей/сети.

### Реальный корпус (13 писем в zakup@ INBOX + КП-508)

Живые ответы поставщиков показали: **чистая цена текстом — меньшинство.** Реальные ответы:
- **отказ** — «Запрашиваемые вами изделия мы не поставляем» (adamant.kz) → цены нет;
- **встречный вопрос** — «Какого объёма холодильник? Со стеклянной дверью?» (medstore.kz) → цены нет;
- **HTML-only** — тело в `text/html`, наш `collect()` берёт только `text/plain` → тело пустое;
- **цена во вложении** — Excel/PDF КП → вне text-парсера;
- иногда — **цена текстом**: «Цена 3 200 000 ₸, срок 3 недели» (тест КП-508).

Вывод: парсер обязан **безопасно возвращать «не нашёл»** на большинстве писем и не выдумывать число.

## Границы (scope)

**В scope:**
- Только **одно-лотовые КП** (ровно 1 `PriceRequestItem`). Много-лотовые не трогаем — там поставщики
  обычно шлют таблицу/Word/Excel → ручной ввод (решение оператора).
- Извлечение **цены** → `responsePrice` (основное).
- Извлечение **срока поставки** → в `responseNote` (бонус, best-effort).

**НЕ в scope (YAGNI):**
- Много-лотовый мап «цена ↔ лот».
- Парс вложений (Excel/PDF).
- Парс модели/№ РУ из ответа (уже известны — предложенная модель лота).
- LLM-фолбэк (отдельный опт-ин пункт бэклога).

## Архитектура

Новый чистый компонент-функция + тонкий хук в существующий приёмник.

### `SupplierReplyPriceParser` (новый, пакет `service`)

Чистая функция без I/O — легко юнит-тестируется на корпусе.

```
Optional<ParsedPrice> parse(String rawBody, Market market)

record ParsedPrice(BigDecimal price, String term /*срок, nullable*/, String matchedSnippet)
```

Алгоритм:
1. **Отрезать цитату нашего письма.** Обрезаем всё от первого маркера цитаты:
   `On … wrote:`, строки, начинающиеся с `>`, `From:`/`Sent:`/`-----Original Message-----`,
   `Отправлено:`/`От кого:`. **Критично** — иначе распарсим числа из НАШЕГО текста
   (дата «Просим ответить до 13.07.2026», «РУ … 030788 — 1 шт»). В КП-508 цитата шла ровно так.
2. **HTML→текст.** Если тело выглядит как HTML (есть теги `<...>`), снять теги, декодировать
   entities (`&nbsp;` `&amp;` `&#…;`), схлопнуть пробелы. Реальные поставщики шлют HTML.
3. **Кандидаты цены по якорям:**
   - валюта рынка: KZ → `₸`, `тенге`, `тг`, `тнг`, `KZT`; RF → `₽`, `руб`, `RUB`;
   - слова-якоря: `цена`, `стоимость`, `сумма`, `за единицу`, `итого`, `составляет`, `прайс`;
   - формат числа: `3 200 000`, `3200000`, `3 200 000,00`, `3.200.000` — пробел/nbsp/точка как
     разделитель тысяч, запятая — десятичная; ≥ 4 значащих цифр (цены на медтехнику — тысячи+).
4. **Отсечь не-цены:** телефоны (`+7…`, `8(…)…`, длинные группы 2-2-2), даты (`dd.mm.yyyy`,
   `dd.mm`), проценты (`%`), количества (`N шт`), числа с тех-единицами (`Гц`, `кГц`, `В`, `мм`,
   `кг`, `°C`), голые годы (`19xx`/`20xx` без разделителей тысяч).
5. **Выбор победителя:** число, максимально близко стоящее к валютному символу/слову-якорю,
   с приоритетом «после `цена/стоимость`» и «перед `₸/тенге`». Если валидного якоря нет —
   **вернуть `Optional.empty()`** (не гадать по случайному числу).
6. **Срок (bonus):** near-match `срок … (\d+)\s*(дн|недел|мес)` или «до dd.mm.yyyy» из НЕ-цитатной
   части → строка в `term`.

### Хук: `MailReceiveService.matchSupplierResponse(...)`

Уже `@Transactional`, уже грузит `PriceRequest`. Добавить после установки `RESPONDED`:

```
if (pr.getItems().size() == 1) {
    PriceRequestItem item = pr.getItems().get(0);
    if (item.getResponsePrice() == null) {              // не затираем ручной/прежний ввод
        parser.parse(body, pr.getMarket()).ifPresent(pp -> {
            item.setResponsePrice(pp.price());
            item.setResponseNote(mark(pp));             // «💡 распознано автоматически — проверьте» + срок
        });
    }
}
```

- `PriceRequest.items` — `@OneToMany cascade ALL` → правка item сохранится с `save(pr)`.
  (Проверить fetch: если LAZY — обращение под тем же tx/OSIV ок; при необходимости — размер по
  `pr.getItems()`.)
- Сеть уже сделана (парсер — чистый CPU), новых tx/§6-нюансов нет.
- Рынок берём из `pr.getMarket()` (у КП рынок проставлен), не из `mailboxMarket` — надёжнее.

## Модель безопасности

- Срабатывает только: **1 item И `responsePrice == null`**.
- Нашёл с якорем → пишем цену + пометку в заметку (оператор видит, что надо проверить).
- Не уверен → `null`, оператор впишет руками. `RESPONDED` ставится всегда (как сейчас).
- Никогда не затираем уже введённую цену.

## Обработка ошибок

- Парсер не бросает — всегда `Optional` (внутри try/catch на всякий, `empty` при сбое).
- Пустое/HTML-битое тело → `empty`.
- Хук обёрнут так, что сбой парса **не ломает** основной путь (RESPONDED + note + InboundEmail
  сохраняются в любом случае).

## Тестирование (TDD, на реальном корпусе)

Юнит-тесты `SupplierReplyPriceParserTest` (чистый, без Spring):
- **КП-508 реальный:** «Цена 3 200 000 ₸, срок 3 недели\n\nOn Fri, Jul 10… wrote:\n> …РУ …030788 — 1 шт\n> Просим ответить до 13.07.2026» → `price=3200000.00`, `term≈"3 недели"`, дата/РУ из цитаты НЕ пойманы.
- **Отказ (adamant):** «Запрашиваемые изделия мы не поставляем» → `empty`.
- **Встречный вопрос (medstore):** «Какого объёма холодильник?» → `empty`.
- **Форматы цены:** `«3200000 тенге»`, `«Стоимость: 3 200 000,00 тг»`, `«цена 3200000»`,
  `«2 950 000 ₸»` → корректное число.
- **Ловушки:** телефон `+7 700 025-88-50` в тексте без цены → `empty` (телефон не выбран);
  цена + телефон вместе → выбрана цена, не телефон.
- **HTML-тело:** `<div>Цена <b>3 200 000</b> ₸</div>` → `3200000.00`.
- **RF-рынок:** «цена 450 000 руб» при `Market.RF` → `450000.00`.

Интеграция (по возможности, GreenMail — расширить существующий round-trip): ответ с ценой на
одно-лотовый КП → после `poll()` у item проставлен `responsePrice`. Если дорого — покрываем юнитом
парсера + ручным клик-тестом (перепослать КП-508-подобный кейс на тест-ящик).

## Открытые вопросы / риски

- Порог «≥4 цифр» может резать дешёвую расходку — приемлемо для MVP (медтехника обычно тысячи+),
  тюним по жалобам.
- HTML-only письма: сейчас `collect()` берёт только `text/plain` → для HTML-only тело всё равно
  пустое ДО парсера. MVP: если `text/plain` пуст, а есть `text/html` — надо, чтобы `collect()`
  отдал html-часть. **Решение:** в `collect()` добавить фолбэк «нет text/plain → взять text/html»,
  парсер уже умеет снимать теги. (Малое расширение, оставляем в scope — иначе парсер бесполезен
  на реальных HTML-ответах.)
