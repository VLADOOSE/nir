# Дизайн: Группировка КП по поставщикам (справочник брендов) — Блок C

**Дата:** 2026-06-25
**Статус:** утверждён, готов к плану
**Контекст:** третий блок расширения АИС. Достраивает поток «Частники» (после блока B): автоподбор поставщиков под брендованные строки заявки и запрос КП по группам. Из roadmap `HANDOFF-АИС-медоборудование.md` (сорсинг + RFQ).

---

## 1. Цель

В частной заявке клиника называет бренд/модель. Блок B дал ручной per-строчный запрос КП. Блок C добавляет **автоподбор поставщиков**: для каждой строки находим, кто из поставщиков возит этот бренд, группируем строки по поставщику и даём запросить КП по группам в один клик. Для этого нужен **справочник «поставщик ↔ бренды»**.

---

## 2. Журнал решений

| # | Развилка | Решение |
|---|----------|---------|
| 1 | Чем матчить | **A — по бренду/производителю** (не по типу, не по модели): у поставщика список брендов, строка заявки матчится по `manufact` |
| 2 | Хранение брендов | `@ElementCollection` строк на `Distributor` (таблица `distributor_brand`) — без отдельной сущности |
| 3 | Строгость матча | **Регистронезависимо, по вхождению/равенству** (case-insensitive). Нечёткий (Mindray↔Майндрей↔Shenzhen Mindray) — улучшение на потом (можно подключить `pg_trgm`) |
| 4 | Редактирование брендов | В карточке/форме поставщика (через существующий update дистрибьютора) |
| 5 | UX группировки | В карточке частной заявки — блок «Подобрать поставщиков»: группы по поставщику + кнопка «Запросить КП» на группу. Строки без поставщика — отдельным списком |
| 6 | Тендеры | Не трогаем — у них своя типовая группировка (`distributor_equipment_type` + `BulkPriceRequestService`) |

## 3. Скоуп блока C

**В скоупе:**
- Справочник брендов поставщика (`distributor_brand`), редактирование в карточке поставщика.
- Сервис подбора: группировка строк частной заявки по поставщикам, у кого есть бренд строки + «строки без поставщика».
- UI в карточке заявки: группы по поставщику + отправка КП по группе (переиспуем `PriceRequest`).
- Ручной per-строчный запрос из блока B остаётся как фолбэк.

**Вне скоупа (далее):**
- Нечёткий матч брендов (`pg_trgm`/синонимы/транслитерация).
- Авто-парсинг почты (блок D).
- Авто-генерация финального КП клиенту (НДС/происхождение).
- Группировка для тендеров (у них уже есть типовая).

## 4. Модель данных

**Бренды поставщика — `@ElementCollection` на `Distributor`:**
```java
@ElementCollection(fetch = FetchType.EAGER)
@CollectionTable(name = "distributor_brand", joinColumns = @JoinColumn(name = "distributor_id"))
@Column(name = "brand", length = 255)
@Builder.Default
private List<String> brands = new ArrayList<>();
```
Таблица в `schema.sql`:
```sql
CREATE TABLE distributor_brand (
    distributor_id BIGINT NOT NULL REFERENCES distributor(id) ON DELETE CASCADE,
    brand          VARCHAR(255) NOT NULL
);
```
Добавить `DROP TABLE IF EXISTS distributor_brand CASCADE;` ПЕРЕД `distributor` в блоке удаления.
Рыночный скоуп — наследуется от родителя `distributor` (бренды — дочерняя коллекция, отдельной колонки `market` не нужно; доступ только через distributor, который скоупится `@Filter`).

## 5. Backend-компоненты

- **`Distributor`** — поле `brands` (см. §4). `DistributorRequest`/`DistributorResponse` — добавить `List<String> brands`. Маппер: `brands` маппится по имени; на `toEntity`/`updateEntity` указать корректное обновление коллекции (MapStruct — заменять список).
- **Редактирование брендов** — через существующий `PUT /api/distributors/{id}` (в `DistributorRequest` приходит `brands`, апдейтит коллекцию). Отдельных эндпоинтов не нужно.
- **`PrivateRequestSourcingService`** (новый, чтобы не раздувать `PrivateRequestService`):
  - `buildSourcing(Long privateRequestId) : SourcingPreview` — для каждой строки заявки (`tender_lot` с `manufact`) находит поставщиков активного рынка, у кого `brands` содержит `manufact` (case-insensitive). Группирует: `List<SupplierGroup{ distributor, lines[] }>` + `List<lines без поставщика>`.
  - Загрузка поставщиков — `distributorRepository.findAll()` (скоупится рынком), фильтрация брендов в Java (поставщиков немного).
- **Эндпоинт:** `GET /api/private-requests/{id}/sourcing` → `SourcingPreviewResponse`. **Отправка по группе — существующий `POST /api/price-requests`** (items = строки группы: `{tenderLotId, medEquipmentId: null, requestedQuantity}`). Новый эндпоинт отправки НЕ нужен.
- **DTO:** `SourcingPreviewResponse { groups: [{ distributor: DistributorResponse, lines: [PrivateRequestLineResponse] }], unmatchedLines: [PrivateRequestLineResponse] }`.

## 6. Frontend

- **Карточка поставщика** (`distributors` страница): редактор брендов — список с добавить/удалить (свободный ввод), сохраняется через update дистрибьютора (поле `brands`).
- **Карточка частной заявки** — новый блок **«Подобрать поставщиков»** (рядом/вместо ручного «Запросить КП»):
  - Грузит `GET /api/private-requests/{id}/sourcing`.
  - По каждому поставщику — карточка: имя + список строк, которые он закрывает + кнопка **«Запросить КП»** (шлёт этому поставщику его строки через `createPriceRequest`).
  - Блок «Строки без поставщика»: перечисление + подсказка «добавьте бренд в карточке поставщика или запросите вручную».
  - Ручной per-строчный выбор (блок B) остаётся ниже как фолбэк.
- Уже запрошенные КП по строке («Запрошен у: …») и блок «Существующие КП» — как в блоке B.

## 7. Обработка ошибок / edge

- Строка без `manufact` (пустой бренд) → в «без поставщика».
- Бренд есть у нескольких поставщиков → строка попадает в группу каждого (оператор сам решает, кому слать).
- Нет поставщиков с этим брендом → строка в «без поставщика».
- Пустой справочник брендов у всех → все строки «без поставщика» (фолбэк — ручной запрос).
- Дубли брендов в списке поставщика → нормализуем (трим, без пустых) при сохранении.

## 8. Тестирование

- **Unit (бэк):** CRUD брендов поставщика (update сохраняет/заменяет список); `buildSourcing` — строка с брендом X группируется к поставщикам, у кого X (case-insensitive), строка без матча → unmatched; строка попадает к нескольким поставщикам.
- **Frontend:** сборка; ручной e2e — в карточке поставщика добавить бренд «Mindray», в карточке частной заявки увидеть группу этого поставщика со строкой Mindray + кнопку, отправить КП по группе.
- **E2E-смоук (рынок KZ):** добавить «Mindray» в бренды «ТОО МедСнаб Казахстан» → открыть `ЧЗ-2026-0001` → блок «Подобрать поставщиков» показывает МедСнаб со строкой «Электрокардиограф BeneHeart R12», «Криосауна» — в «без поставщика» → запросить КП по группе → появляется в «Существующих КП».
