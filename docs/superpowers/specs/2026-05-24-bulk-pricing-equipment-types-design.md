# Дизайн: массовый подбор оборудования + справочник типов + специализация дистрибьюторов

**Дата:** 2026-05-24
**Контекст:** АИС учёта участия в тендерах на медоборудование (ООО «Регион-Мед»)

## Зачем

Текущая система даёт оператору только атомарные операции: «подобрать оборудование под один лот», «запросить КП у одного дистрибьютора по одной позиции». Для тендера на 5 лотов это означает 5 нажатий «Подобрать» + ~15 нажатий «Запросить КП» (по 3 дистрибьютора на лот) + 15 писем в почте. Это не помощь, а ручная работа в форме интерфейса.

Также сейчас:
- При выборе оборудования к позиции заявки показывается весь каталог без фильтра по типу.
- Связи «дистрибьютор ↔ что он может поставить» в системе нет, оператор её помнит сам.
- Типы оборудования (УЗИ, Рентген, ИВЛ, Монитор) — захардкоженный список в формах фронта.

## Что меняем (краткая сводка решений)

1. **Тип оборудования = отдельный справочник.** Новая таблица `equipment_type`, FK из `med_equipment` и `tender_lot`. Админ может добавлять новые типы через UI.
2. **Специализация дистрибьютора = M2M-связь с типами.** Пустая специализация = универсал (попадает во все подборы).
3. **Структура `price_request` пересобрана.** Запрос = заголовок (дистрибьютор + общий статус) + список позиций (`price_request_item`). Одно письмо может содержать несколько позиций по одному тендеру.
4. **Кнопка «Запросить КП по всему тендеру»** — на странице тендера, фаза разведки цен.
5. **Кнопка «Собрать заявку из КП»** — на странице черновика заявки, фаза формирования предложения.
6. **Карточка оборудования** — модал поверх каталога с потенциальными поставщиками, историей запросов и рейтингом дистрибьюторов.
7. **Фильтр по типу** при выборе оборудования к позиции заявки.

## Модель данных

### Новые таблицы

```sql
CREATE TABLE equipment_type (
    id   BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE distributor_equipment_type (
    distributor_id    BIGINT NOT NULL REFERENCES distributor(id) ON DELETE CASCADE,
    equipment_type_id BIGINT NOT NULL REFERENCES equipment_type(id) ON DELETE CASCADE,
    PRIMARY KEY (distributor_id, equipment_type_id)
);

CREATE TABLE price_request_item (
    id                 BIGSERIAL PRIMARY KEY,
    price_request_id   BIGINT NOT NULL REFERENCES price_request(id) ON DELETE CASCADE,
    tender_lot_id      BIGINT NOT NULL REFERENCES tender_lot(id),
    med_equipment_id   BIGINT NOT NULL REFERENCES med_equipment(id),
    requested_quantity INTEGER NOT NULL,
    response_price     NUMERIC(15, 2),
    response_note      TEXT
);
```

### Изменения существующих таблиц

```sql
-- med_equipment: equip_type строка → FK
ALTER TABLE med_equipment DROP COLUMN equip_type;
ALTER TABLE med_equipment ADD COLUMN equip_type_id BIGINT REFERENCES equipment_type(id);

-- tender_lot: то же
ALTER TABLE tender_lot DROP COLUMN equip_type;
ALTER TABLE tender_lot ADD COLUMN equip_type_id BIGINT REFERENCES equipment_type(id);

-- price_request: убрать "позиционные" поля (переехали в item), добавить tender_id
ALTER TABLE price_request DROP COLUMN tender_lot_id;
ALTER TABLE price_request DROP COLUMN med_equip_id;
ALTER TABLE price_request DROP COLUMN response_price;
ALTER TABLE price_request DROP COLUMN response_note;
ALTER TABLE price_request ADD COLUMN tender_id BIGINT NOT NULL REFERENCES tender(id);
ALTER TABLE price_request ADD COLUMN note TEXT;
```

Поле `status` в `price_request` остаётся (общий статус группы). Значения: `CREATED` → `SENT` → `RESPONDED` → `CLOSED`. Per-item статус не нужен — отсутствие `response_price` сам по себе означает «по этой позиции ответа нет».

### Сидинг типов и специализаций

`DataInitializer` при пустой `equipment_type` заполняет: `УЗИ`, `Рентген`, `ИВЛ`, `Монитор`.

В `data.sql` обновить:
- `med_equipment.equip_type_id` — указать FK вместо строки.
- Существующих дистрибьюторов оставить с пустой специализацией (= универсалы), пока админ не настроит вручную.

### Backend artifacts

- Entity: `EquipmentType`, `PriceRequestItem`. У `Distributor` — `@ManyToMany List<EquipmentType> equipmentTypes`. У `MedEquipment` и `TenderLot` — `@ManyToOne EquipmentType equipmentType`.
- Repository: `EquipmentTypeRepository`, `PriceRequestItemRepository`, новый метод `MedEquipmentRepository.findByEquipTypeId(Long)`, метод в `DistributorRepository` поиска «по типу или универсал» (см. секцию «Логика подбора»).
- Service: `EquipmentTypeService`, `BulkPriceRequestService` (новый — оркестрирует подбор + создание запросов + отправку писем), `ApplyAutoFillService` (новый — автозаполнение заявки из принятых КП).
- DTO: `EquipmentTypeRequest/Response`, `PriceRequestItemRequest/Response`, обновлённые `MedEquipmentResponse` / `TenderLotResponse` / `DistributorResponse` (с типом / специализацией).
- Controller: `EquipmentTypeController`, новые endpoints в `TenderController` (или новый `BulkPriceController`) и `ActivityApplyController` для двух кнопок.

### Защита от удаления

- Нельзя удалить `equipment_type`, если он используется в `med_equipment` или `tender_lot` (по образцу существующих defensive checks).

## Логика подбора и группировки

### Подходящие дистрибьюторы для типа T
```
SELECT d FROM distributor d
LEFT JOIN distributor_equipment_type det ON det.distributor_id = d.id
WHERE det.equipment_type_id = :typeId
   OR NOT EXISTS (SELECT 1 FROM distributor_equipment_type WHERE distributor_id = d.id)
```
Универсалы (без записей в M2M) попадают во все подборы.

### Подходящее оборудование для лота L
Существующий `MedEquipmentRepository.findMatchingEquipment`, плюс жёсткий фильтр по `equip_type_id = L.equip_type_id` (раньше фильтр шёл по строке).

### Группировка для массового подбора по тендеру T
```
для каждого лота L в T:
    подходящие_модели = findMatching(L)
    подходящие_дистрибьюторы = findDistributorsForType(L.equip_type_id)
    для каждой пары (модель M, дистрибьютор D):
        группа[D].add(L, M)
```
Результат — словарь `Distributor → список (TenderLot, MedEquipment[])`. В UI каждый Distributor становится карточкой со списком лотов и моделей.

### Автосборка заявки из КП
Для тендера `T` и черновика заявки `A`:
```
для каждого лота L в T:
    кандидаты = price_request_item где tender_lot_id = L.id AND response_price IS NOT NULL
    если есть:
        лучший = min(response_price) среди кандидатов
        создать apply_item(apply=A, lot=L, equipment=лучший.med_equipment,
                          distributor=лучший.price_request.distributor,
                          offered_cost=лучший.response_price, quantity=L.quantity)
```
Если по лоту нет ни одного ответа — пропускаем (оператор увидит «по лоту X нет подобранных предложений»).

## UI

### Справочник типов оборудования (`/equipment-types`, только ADMIN)
Простой CRUD на одно поле `name`. Пункт в сайдбаре под «Каталог». Защита от удаления при использовании.

### Дистрибьюторы — форма
Новый блок «Специализация»:
- Чекбокс «Все типы» — отмечен и disabled по умолчанию.
- Под ним чекбоксы со всеми типами из `/api/equipment-types`.
- Поведение: как только оператор отмечает любой конкретный тип, «Все типы» автоматически снимается и становится снова кликабельной. Как только все конкретные сняты, «Все типы» автоматически отмечается и снова disabled.
- В БД сохраняется: список выбранных типов (пустой = универсал).

В таблице дистрибьюторов — новая колонка «Специализация»: либо плашка «Все типы», либо теги.

### Оборудование — модал-карточка
Клик по строке в таблице открывает sidebar справа с тремя секциями:

**1. Характеристики** — все поля сущности read-only.

**2. Потенциальные поставщики** — список дистрибьюторов с подходящей специализацией (универсалы тоже). Для каждого: имя, контакт, email, кнопка «Запросить КП» (открывает существующую форму создания PR на этого дистрибьютора с предзаполненной позицией).

**3. История запросов** — три блока:

- **Сводка по ответам:**
  > Запрашивали: N раз у K дистрибьюторов
  > Цены ответов: от X ₽ до Y ₽, средняя Z ₽

- **Рейтинг дистрибьюторов** (сортировка по средней цене ASC):
  > • МедТехСнаб — 5 ответов, средняя 820 000 ₽ ← лучший
  > • Глобал Медикал — 3 ответа, средняя 875 000 ₽

- **Таблица истории** (по убыванию даты):
  | Дата | Дистрибьютор | Тендер | Кол-во | Цена ответа | Статус запроса |

### Лоты — фильтр оборудования по типу
В форме «Добавить позицию к заявке» (`applies.component`): когда лот выбран, select оборудования показывает только модели с `equip_type_id` равным типу лота. Под select-ом подпись «Показаны только аппараты типа УЗИ» (заменяется на актуальный тип). Если у лота тип не указан — fallback на полный каталог.

### Тендер — кнопка «Запросить КП по всему тендеру»
На странице деталей тендера (`tenders.component` в режиме selectedTender), рядом с «Добавить лот». Активна когда есть хотя бы один лот.

Открывает экран подбора (модал на весь экран или route `/tenders/:id/bulk-price`):
- Сверху — сводный список лотов (свёрнут по умолчанию).
- Дальше — группы по дистрибьюторам. Каждая группа = карточка:
  - Заголовок: имя дистрибьютора, контакт, email, типы которые он покрывает.
  - Внутри — список лотов которые он может закрыть, под каждым лотом — подходящие модели с чекбоксами. По умолчанию отмечены все, кроме тех что выходят за `max_cost` лота (помечены ⚠ и сняты).
  - Снизу группы — кнопка «Превью письма» (раскрывает текст для правки) и «Отправить КП».
- Отдельная секция «Лоты без вариантов» — два кейса:
  - Нет подходящих моделей в каталоге (`findMatchingEquipment` вернул пусто) → текст «Нет подходящего оборудования. Добавьте в каталог или ослабьте требования лота».
  - Модели есть, но нет дистрибьюторов для типа (даже среди универсалов) → текст «Нет дистрибьюторов с типом X. Настройте специализацию или добавьте дистрибьютора».

При нажатии «Отправить КП» в группе:
1. POST на новый endpoint `/api/bulk-price/send` с телом `{ tenderId, distributorId, items: [{tenderLotId, medEquipmentId, quantity}] }`.
2. Бэк создаёт `price_request` (status=SENT, sent_at=now) + `price_request_item` на каждую отмеченную позицию.
3. Бэк генерирует email с шаблоном:
   ```
   По тендеру №<tender_number> просим предоставить КП на следующие позиции:

   | Лот | Наименование | Производитель | Кол-во |
   | 1   | SonoAce R7   | Samsung       | 2      |
   | 1   | Vivid T8     | GE            | 2      |
   | 3   | Luminos dRF  | Siemens       | 1      |

   Просим указать: цену за единицу, сроки поставки, условия оплаты, гарантию.

   С уважением, ООО «Регион-Мед»
   ```
4. Отправляет через `JavaMailSender` (`EmailService.sendEmail`).
5. Возвращает 200 → группа в UI помечается «Отправлено», кнопка disabled.

Если оператор передумал в другой группе — она просто остаётся не нажатой. **В БД ничего не создаётся для неотправленных групп.**

### Тендер — секция «Запросы КП»
Текущий плоский список позиций → переделать в карточки по `price_request`:
- Заголовок карточки: дистрибьютор, статус группы (CREATED/SENT/RESPONDED/CLOSED), дата отправки.
- Внутри — список позиций (`price_request_item`): лот, модель, кол-во. Если есть `response_price` — показывается.
- Кнопка «Внести ответы» — раскрывает форму для каждой позиции: поле «Цена ответа», «Заметка». Сохранение обновляет item-ы.
- Переходы статуса:
  - `CREATED → SENT` — автоматически при отправке письма из массового подбора.
  - `SENT → RESPONDED` — автоматически когда оператор сохранит «Цену ответа» хотя бы по одной позиции.
  - `RESPONDED → CLOSED` — вручную через кнопку «Закрыть запрос» на карточке (после того как оператор использовал ответы в заявке либо отказался).

### Заявка — кнопка «Собрать из КП»
На странице деталей черновика заявки (`applies.component`, selectedApply со статусом DRAFT), рядом с «Добавить позицию». Кнопка активна только если по тендеру этой заявки есть `price_request_item` с непустой `response_price`.

При нажатии:
1. POST `/api/applies/{id}/auto-fill`.
2. Бэк выполняет логику автосборки (см. выше).
3. Возвращает обновлённый список items и список лотов, по которым не было ответов (для информирования оператора).
4. UI показывает уведомление: «Добавлено N позиций, по лотам X, Y нет КП с ответом».

Если для одного лота уже есть позиция в заявке — пропускается (не дублируем). Оператор удалит вручную если хочет перезаполнить.

## Что НЕ делаем (YAGNI)

- **Сравнение полученных КП в едином экране** (`/tenders/:id/compare`). Польза есть, но карточка оборудования с рейтингом уже даёт основное.
- **Автоматическое напоминание о просрочке КП** (если дистрибьютор не ответил за N дней). Email-уведомления и таймеры — отдельная история.
- **Ручное редактирование сгенерированного письма с сохранением шаблона.** Превью + одноразовая правка перед отправкой — этого достаточно.
- **Статус на каждую позицию `price_request_item`** (ACCEPTED/REJECTED). Достаточно факта наличия `response_price`.
- **ML-рекомендация «лучший дистрибьютор»** для лота. Рейтинг по средней цене из карточки оборудования — простой baseline.
- **Загрузка прайс-листа от дистрибьютора файлом** (Excel/CSV). Дистрибьюторы отвечают свободным текстом в почте.

## План миграции данных

1. Добавить новые таблицы (`equipment_type`, `distributor_equipment_type`, `price_request_item`) — DDL в `schema.sql`.
2. Изменить `med_equipment`, `tender_lot`, `price_request` — DDL.
3. `DataInitializer` заполняет `equipment_type` четырьмя дефолтными типами.
4. Обновить `data.sql`: для seed-оборудования и seed-лотов выставить `equip_type_id` (через подзапрос `SELECT id FROM equipment_type WHERE name='УЗИ'`).
5. У сидируемых дистрибьюторов специализацию НЕ выставляем — будут универсалами, что не ломает существующие сценарии.

`spring.jpa.hibernate.ddl-auto: none` + dev-режим с пересозданием по `schema.sql` означает, что миграция «через DDL+DML» — это просто новые версии файлов. Production-миграция за рамками.

## Безопасность и разграничение ролей

- `/api/equipment-types` — POST/PUT/DELETE: `@PreAuthorize hasRole('ADMIN')`. GET — для всех аутентифицированных (нужен для выпадашек).
- `distributor.equipmentTypes` (специализация) — редактируется только ADMIN (это часть справочника).
- Кнопки «Запросить КП по всему тендеру» и «Собрать из КП» — доступны USER (это оперативные действия).
- Карточка оборудования — доступна USER (только просмотр).
