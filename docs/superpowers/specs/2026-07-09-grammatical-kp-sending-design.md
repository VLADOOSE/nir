# Дизайн: грамотная отправка КП + приём ответов (часть 2)

- **Дата:** 2026-07-09
- **Статус:** согласовано (brainstorming), готов к writing-plans
- **Рынок:** прежде всего KZ (West-Med), механика рынко-нейтральна
- **Тема-родитель:** «грамотно отправлять КП и получать ответы» (часть 2 из 3; ч.1 «подбор по виду МИ» — закрыта)

## 1. Контекст и проблема

Канал отправки КП уже есть (`PriceRequestSendService.send` → `KpEmailComposer.compose` →
`EmailService.sendEmail`, единый путь для всех флоу; токен `[КП-<id>]` в теме; статусы
CREATED→SENT→RESPONDED→ACCEPTED/CLOSED; приём ответов `MailReceiveService.poll` матчит по токену).
Но «отправка» пока не «грамотная»:

1. **Тело письма ломается на реальных данных.** Приветствие `Уважаемый(ая) <фамилия> <имя>!` — у 20
   исследованных KZ-дистрибьюторов (V9) контактного лица нет → выходит «Уважаемый(ая)  !». Нет явной
   инструкции «отвечайте на zakup@», нет reply-to.
2. **Нельзя увидеть/поправить текст перед отправкой.** Оператор шлёт вслепую.
3. **Нет реального round-trip.** SMTP по умолчанию — MailHog-фейк; отправки через боевой
   `zakup@westmed.kz` нет. При тестовой отправке на сам `zakup@` наш же исходящий КП попадёт в тот же
   ящик и поллер по токену ложно пометит его RESPONDED.
4. **Слабый UX статусов.** Нет «переслать» (для NO_EMAIL/SEND_FAILED после правки email), нет
   «проверить ответы» кнопкой.

## 2. Границы

Родительский блок = 3 части: ч.1 подбор по виду МИ (**закрыта**); **ч.2 [ЭТОТ СПЕК] грамотная
отправка + приём**; ч.3 структурный разбор ответа поставщика (цена/срок/модель → `PriceRequestItem`)
— **вне этого спека** (§10 follow-up).

### Что НЕ трогаем (уже готово)
- Единый канал `PriceRequestSendService` (дедуп поставщиков, запись через `PriceRequestWriter`,
  письмо после коммита, гард рынка, `SendResult{emailSent, reason}`).
- Токен `[КП-<id>]` (`KpToken.subjectToken/parse`), матч ответа по токену в `MailReceiveService`.
- Секция «Запросы КП» в карточке тендера (бейджи статусов, «Принять КП», «Закрыть запрос»).
- Тело письма в части позиций (лот/модель/РУ/ТЗ/дедлайн/ссылка на объявление/запрос цены-сроков) —
  оставляем, дорабатываем только приветствие/подпись.

## 3. Согласованные решения (brainstorming)

| Решение | Выбор |
|---|---|
| Ядро захода | качество письма + редактируемое превью + реальный round-trip + UX статусов |
| Reply-To | = отправляющий ящик (zakup@westmed.kz), чтобы ответы падали в опрашиваемый ящик |
| Override тела | единый на выбранную группу; **токен `[КП-id]` всегда серверный**, не редактируется |

| Разбор цены из ответа | вне ч.2 — это ч.3 |

## 4. Качество письма (`KpEmailComposer`)

- **Приветствие:** есть ФИО контакта (`lastName`/`firstName`) → «Уважаемый(ая) <Фамилия> <Имя>!»
  (как сейчас); нет ФИО → **«Здравствуйте!»**. Далее первое предложение обращается к компании:
  «Компания <companyShortName> просит …» (уже есть) — при отсутствии лица этого достаточно.
- **Подпись:** после «С уважением, <companyShortName>» добавить строку-инструкцию:
  «Ответ на этот запрос просим направить в ответном письме (Reply) — он поступит на zakup@westmed.kz.»
  Адрес ответа — из конфига (§6), не хардкод в тексте (fallback — `spring.mail.username`).
- Порядок/содержание позиций — без изменений.

## 5. Reply-To и UTF-8 (`EmailService`)

`EmailService.sendEmail` переводится с `SimpleMailMessage` на **`MimeMessage` + `MimeMessageHelper`**
(`spring-boot-starter-mail` уже подключён):
- `helper.setFrom(fromAddress)`, `helper.setReplyTo(replyToAddress)`, UTF-8, `setText(body, false)`
  (plain-text).
- `replyToAddress` = `${mail.kp.reply-to:${spring.mail.username}}` — по умолчанию = ящик отправки.
- Сигнатура `sendEmail(to, subject, body)` сохраняется (внутренняя реализация меняется) — вызовы
  `PriceRequestSendService`/остальные не трогаем. `isConfigured()` — как есть.

## 6. Отправка через zakup@ (конфиг, без изменения кода отправки)

Боевой SMTP включается только через env (по образцу живого IMAP §5 CLAUDE.md), дефолт — MailHog:
```
MAIL_HOST=smtp.mail.ru MAIL_PORT=465 MAIL_USERNAME=zakup@westmed.kz \
MAIL_PASSWORD="$(cat /tmp/imap.pass)" \
MAIL_SMTP_SSL=true MAIL_SMTP_AUTH=true \
./gradlew bootRun
```
- `application.yaml`: `spring.mail.properties.mail.smtp.auth: ${MAIL_SMTP_AUTH:false}`,
  `mail.smtp.ssl.enable: ${MAIL_SMTP_SSL:false}` (MailHog: обе false; Mail.ru 465: обе true).
  Таймауты оставить.
- Ничего не эхо-печатать из `/tmp/*.pass` (только `$(cat …)`).

## 7. Round-trip приём: пропуск «своих» писем (`MailReceiveService`)

При тестовой отправке на сам `zakup@westmed.kz` наш исходящий КП вернётся в тот же ящик и поллер по
токену `[КП-id]` ложно пометит его RESPONDED. Фикс: **если From письма == адрес отправки
(`spring.mail.username`, нормализованный lower/trim), письмо НЕ классифицируется как
`SUPPLIER_RESPONSE`** — помечается SEEN и пропускается (idempotent). Реальный ответ (From =
поставщик, тот же токен) матчится как прежде → RESPONDED.
- Сравнение по адресной части (без display-name), регистронезависимо.
- Гард `since-minutes` и рыночная обвязка (§6 CLAUDE.md) — без изменений.

## 8. Редактируемое превью + override в /send

- **`POST /api/price-requests/preview`** body `{tenderId, distributorIds[], items[]}` →
  `{subject, body}`. Компонует человеческую часть тем же `KpEmailComposer` на **несохранённом**
  черновом `PriceRequest` (id ещё нет → тема БЕЗ токена; фронт показывает пояснение «№ КП и метка
  [КП-…] будут присвоены при отправке»). Первый выбранный дистрибьютор — как образец обращения.
  `@PreAuthorize("hasRole('ADMIN')")`.
- **`PriceRequestSendService.send` расширяется** необязательными `subjectOverride`, `bodyOverride`:
  - `body` = `bodyOverride` (если задан и не пуст) иначе `composer.compose(pr).body()`.
  - `subject` = `KpToken.subjectToken(pr.id)` + пробел + (человеческая часть `subjectOverride` если
    задан иначе дефолтная тема без токена). **Токен подставляется сервером всегда** — даже если
    оператор стёр его в превью. Матч ответа по токену гарантирован.
  - Один и тот же override уходит всем выбранным поставщикам группы (персонализация имени —
    follow-up; у дистрибьюторов лиц нет).
- `POST /api/price-requests/send` (контроллер) — принять `subjectOverride`/`bodyOverride` из тела.

## 9. UX статусов в карточке (`tenders.component.ts`, секция «Запросы КП» + КП-панель)

- **Диалог превью:** «Отправить запросы (N)» в КП-панели → сперва `POST /preview` → модалка с полями
  `subject` (input) + `body` (textarea), редактируемые → «Отправить» шлёт `/send` с override →
  тост по `SendResult` (как сейчас) + перезагрузка. «Отмена» закрывает без отправки.
- **Resend:** `POST /api/price-requests/{id}/resend` — перекомпоновать и переслать письмо
  СУЩЕСТВУЮЩЕГО PR (не создаёт новый; гард рынка; тот же токен по его id) → `SendResult`. Кнопка
  «↻ Переслать» в карточке КП — показывается для статусов, где письмо не ушло/можно повторить
  (SENT/CREATED; полезно после правки email поставщика). Обновляет `sentAt`.
- **«Проверить ответы»:** кнопка над секцией «Запросы КП» → `POST /api/inbound/poll` затем
  `loadPriceRequests()` → пришедшие ответы станут RESPONDED (жёлтый бейдж). Показать `sentAt` и
  пометку «письмо ушло»/«письмо не ушло» на карточке КП.
- Всё под существующими паттернами (`cdr.detectChanges()`, `NotificationService`).

## 10. Тестирование

- **`KpEmailComposerTest`** — приветствие с ФИО и без (Здравствуйте!); подпись содержит инструкцию про
  ответ; позиции рендерятся; override-независимая человеческая часть темы.
- **`EmailServiceTest` / round-trip на GreenMail** (встроенный SMTP+IMAP, как
  `MailReceiveServiceIntegrationTest`): отправка КП с Reply-To → приём ответа с токеном `[КП-id]` от
  «поставщика» → статус RESPONDED; **кейс «своё письмо» (From==наш адрес) не помечается RESPONDED**.
- **preview vs send** — человеческая часть тела из `/preview` совпадает с реально отправленной (при
  пустом override); при заданном `bodyOverride` уходит именно он, но тема сохраняет серверный токен.
- **`resend`** — не создаёт новый PR, перешлёт письмо того же id, гард чужого рынка (403/404).
- Гейт «зелёного»: только 2 пред-существующих `ApplyAutoFillServiceTest`. Фронт — `npm run build`.
- **Живая проверка (Playwright + реальный zakup@westmed.kz):** превью → правка → отправка на
  тестовый адрес (сам zakup@) → «проверить ответы» → убедиться, что СВОЁ письмо не стало RESPONDED;
  затем ответить с другого адреса с тем же токеном → RESPONDED. (Живой SMTP/IMAP — под env, вручную.)
