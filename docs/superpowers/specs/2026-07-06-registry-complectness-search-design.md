# Поиск по комплектности аппаратов (аксессуарные лоты → родительское РУ)

**Дата:** 2026-07-06. **Статус:** одобрено пользователем (live-путь через ndda-клиент, триггер — кнопка в панели «Реестр», финал — component-adopt до предложенной модели лота).

## Проблема (проверено живьём 2026-07-06)

Лот 3088 «Электрод» тендера 17279420-1, ТЗ: *«Резиновые пластинки для аппарата электрофореза "Элэскулап", размеры 55×80 мм»*, `manufact` пуст. Текущий реестр-матч (триграмма по названию) цепляется за слово «электрод» → выдаёт ЭКГ/хирургические электроды из Китая/Германии/Индии — «в корне не то», России нет.

Искомого нет **отдельной записью** реестра: электрод зарегистрирован **в комплектности родительского аппарата** «АСЭтМ-01/6 ЭЛЭСКУЛАП» (ООО «Мед ТеКо», **РОССИЯ**, РК МИ (МТ)-0№027673, ndda id 178624). Его допуск в РК покрыт РУ аппарата. Матч по названию до комплектности не дотягивается никогда.

## Находка — API комплектности (публичный, без auth)

`GET {base}/RegisterService/MtComplectList?registerId={nddaId}` → JSON-массив компонентов. Живая форма (снята 2026-07-06 для id 178624), значимые поля элемента:

- `productName` — текст компонента (блоб с размерами), напр. *«4.Электроды силиконовые электропроводящие, мм: - 25 х 30 – 2 шт.; - 55 х 80 – 2 шт.; - 100 х 120 – 2 шт.»*;
- `component` — тип: «основной блок» / «комплектующие»;
- `producerName` — производитель компонента (*ООО «Мед ТеКо»*);
- `countryName` — страна (*Россия*);
- `partNumber` — порядковый номер в составе (int).
- (есть `*Kz`-дубли и `model`/`endDate` — не используем.)

`base = https://oldregister.ndda.kz/register-backend` (тот же, что у детали РУ). `registerId` — внутренний ndda id (как у `MtMainGetById`), НЕ № РУ → резолв через существующий `NddaClient.resolveId(regNumber)`.

Масштаб: комплектность осмысленна у ~2524 записей типа «(МТ)» (аппаратура); у 3011 «(ИМН)» её обычно нет.

## Решение

Кнопка в панели «Реестр» запускает live-цепочку: извлечь имя аппарата из лота → найти аппараты в реестре → резолв ndda id → live-фетч комплектности → сматчить компоненты → показать → «Взять в работу» на компоненте (component-adopt). Massовый скрейп 2.5k комплектностей НЕ делаем.

### 1. Пайплайн (`ComplectSearchService.search(lotId, termOverride)`)

1. Гард: лот существует, рынок лота == текущий (иначе 404, паттерн adopt); только KZ (кнопка `isKz()`).
2. **Термин**: `termOverride` (если оператор задал) иначе `ComplectTermExtractor.extract(lot)` (см. §2).
3. Пустой термин → ответ с `term=null`, `apparatuses=[]` (UI: «уточните название аппарата»). Сети нет.
4. **Аппараты-кандидаты**: `MedRegistryRepository.findApparatusByTerm(term, K)` — `WHERE reg_number LIKE '%(МТ)%' AND :term <% name ORDER BY word_similarity(:term,name) DESC LIMIT K` (K=3, индексо-дружелюбный `<%`).
5. Для каждого кандидата (сеть ВНЕ транзакции):
   - ndda id: `reg.nddaId` если закеширован, иначе `NddaClient.resolveId(regNumber)`; null → пропускаем кандидата;
   - `NddaClient.fetchComplectList(nddaId)` → компоненты; `ComplectWriter.cache(regNumber, components)` (@Transactional отдельный бин) в `registry_component`.
6. **Матч компонентов**: `ComplectComponentMatcher.score(lot, component.productName)` — токены айтема лота (название+ТЗ, generic-слова здесь — дискриминаторы: «электрод/силиконов/55/80») против `productName` (`word_similarity`/покрытие токенов). Компоненты сортируются по score убыв.; показываем ВСЕ компоненты аппарата (оператор подтвердил аппарат — пусть выберет), лучший сверху.
7. Ответ: аппараты (лучший по совпадению термина первым), у каждого — отсортированные компоненты.

Латентность: до 2×K live-запросов первый раз (~3–8 с); ndda id и комплектность кешируются → повтор мгновенный. Потому по кнопке.

### 2. Извлечение термина — `ComplectTermExtractor` (эвристика, самое хрупкое)

Источник: `lot.equipName + " " + (lot.requiredSpec | lot.manufact)`. Приоритет:
1. **Закавыченная фраза** — regex по « », " ", „ ", ' ' → сильнейший кандидат («Элэскулап»).
2. **Паттерн «(для|к) аппарат(а|у)/систем(ы|е) <X>»** → следующий значимый токен/фраза.
3. Иначе — **отличительный токен**: токенизация с расширенным стоп-листом (канцелярит `LotQueryTokenizer` + generic-мед: электрод\*, пластин\*, резинов\*, силиконов\*, размер\*, мм, шт, комплект, аппарат, для, к, и…) → самый длинный оставшийся токен.

Результат — строка-догадка (может быть пустой). НЕ жёсткая: UI показывает её в **редактируемом поле**; оператор правит/вводит и жмёт «Искать» → `termOverride`. Так «система сама находит», но есть ручной штурвал на промахи эвристики.

### 3. Бэкенд

- **`NddaClient.fetchComplectList(long nddaId) : List<NddaComplectItemDto>`** + DTO `NddaComplectItemDto {productName, component, producerName, countryName, partNumber}` (`@JsonIgnoreProperties(ignoreUnknown=true)`). Ошибки сети/не-200/кривой JSON → `UpstreamException` (как остальные методы клиента). Живая форма закреплена фикстурой `NddaHttpClientTest`.
- **Flyway V6 `registry_component`** (общая таблица, без market): `id BIGSERIAL PK, reg_number TEXT (ссылка на med_registry.reg_number, аппарат), part_number INT, product_name TEXT, component TEXT, producer TEXT, country TEXT, fetched_at TIMESTAMPTZ`, `UNIQUE(reg_number, part_number)`. Entity `RegistryComponent`. Кеш апсертится при фетче; adopt читает из кеша (server-authoritative, клиенту медданные не доверяем; портал не долбим повторно). GIN-trgm на `product_name` не обязателен (матч в Java по ≤~15 строкам аппарата).
- **`POST /api/lots/{id}/complect-search`** (query `term` опц.) → `ComplectSearchResponse {term, apparatuses: [{regNumber, name, producer, country, components: [{partNumber, productName, component, producer, country, score}]}]}`. Сеть вне tx; кеш — `ComplectWriter` (@Transactional отдельный бин, паттерн §6 / TechSpecWriter).
- **`POST /api/lots/{id}/adopt-component {regNumber, partNumber}`** — component-aware adopt (ОТДЕЛЬНЫЙ метод от `adoptForLot`, тот берёт имя из записи-аппарата):
  - гард рынка лота (как adopt); аппарат `MedRegistry` по `regNumber` (404 если нет); компонент из `registry_component` по `(regNumber, partNumber)` (404 если не закеширован — сперва complect-search);
  - reuse-или-create `MedEquipment`: `findFirstByRegistrationRegNumberAndNameIgnoreCase(regNumber, name)` иначе create — **name = trim255(component.productName)** (электрод), **manufact = component.producer|«не указан»**, **registration = РУ аппарата** (допуск компонента покрыт им), `REGISTERED`, **spec = component.productName** (размеры), market = `MarketContext`;
  - `lot.setProposedEquipment(eq)`. Дальше — существующий КП-флоу.

`med_registry`/`registry_component` — общие (без market): рыночные фильтр/штамп не участвуют; рыночный гард — только в adopt-component (через лот).

### 4. UI — панель «Реестр» (`tenders.component.ts`)

Под таблицей кандидатов — кнопка **«Комплектность аппаратов»** (KZ, для лота без хорошего матча полезнее всего, но доступна всегда). По нажатию:
- строка: **поле термина** (авто-подставлена догадка, напр. «Элэскулап») + кнопка «Искать»;
- «Ищем в комплектности аппаратов…» (может быть несколько секунд);
- блок результатов: карточка аппарата (РУ, производитель, **Страна**) → список компонентов (текст, тип, страна, score — лучший сверху) с кнопкой **«Взять в работу»** на компоненте → бейдж «Предложено… РУ ✓» + КП-панель (как у обычного adopt);
- пусто → «Аппарат не найден — уточните название в поле выше».
Фронт-кеш результата на объекте лота (повторное открытие без запроса).

### 5. Ошибки и тесты

- Не-KZ/несуществующий лот → 404/кнопка скрыта; портал недоступен → 502 (кеш не пишем, retry кнопкой); термин пуст/аппарат не найден → 200 с пустым списком + подсказка; резолв ndda id null → кандидат пропущен.
- **Тесты**: `NddaHttpClientTest.fetchComplectList` (фикстура живого ответа id 178624 — стаб JDK HttpServer); `ComplectTermExtractorTest` (эвристика: «Резиновые пластинки для аппарата электрофореза "Элэскулап" 55×80» → «Элэскулап»; чистый generic → пусто); `ComplectComponentMatcherTest` (компонент «силиконовые электроды 55×80» ранжируется выше «токопроводящие терапевтические»); `ComplectSearchServiceTest` (@SpringBootTest, mock `NddaClient`: find→resolve→fetch→cache→match; повтор из кеша без сети; портал 502 → кеш пуст); `ComplectAdoptTest` (component-adopt: позиция с РУ аппарата + производителем компонента + spec=текст; reuse не плодит дубль; чужой рынок → 404). Фронт — `npm run build` + Playwright на лоте 3088 (кнопка → «Элэскулап» → аппарат ЭЛЭСКУЛАП РФ → компонент #4 → «Взять в работу» → предложенная модель).
- Гейт бэка: только 2 известных `ApplyAutoFillServiceTest`.

## Вне скоупа

- Bulk-импорт всей комплектности (2.5k аппаратов) в БД + матч по компонентам в общем токен-поиске (обсуждали, отклонено в пользу live).
- Точное вычленение под-размера (55×80) из блоба компонента — берём компонент целиком (он и зарегистрирован как единый пункт со всеми размерами).
- Автотриггер без кнопки / fallback при слабом матче.
- Отправка КП по component-adopt — существующий флоу уже есть; отдельно обсуждается по просьбе пользователя.
- Транслит бренда (Mindray↔Майндрей) для термина — как и в основном матче, отдельный бэклог.
