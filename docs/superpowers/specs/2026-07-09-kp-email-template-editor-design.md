# Дизайн: редактор шаблона письма КП

- **Дата:** 2026-07-09
- **Статус:** согласовано (brainstorming), готов к writing-plans
- **Рынок:** оба (RF/KZ) — отдельный шаблон на рынок
- **Мотив:** оператор спросил «как поменять шаблон письма». Разовая правка в диалоге превью уже есть (ч.2), но она не сохраняется. Нужен сохраняемый дефолтный шаблон, редактируемый из интерфейса.

## 1. Контекст и проблема

Тело/тема письма КП строятся в единственном месте — `KpEmailComposer.buildBody`/`humanSubject`. Это НЕ плоский текст: цикл по позициям, условные РУ/ТЗ, ссылки на объявление, разное слово «тендер»/«заявка», брендинг и реестр по рынку. Сейчас поменять дефолт можно только правкой Java-кода. Оператору нужен редактор в UI, но нельзя терять структурную автоподстановку (список позиций и т.п.).

Решение: шаблон = текст с **плейсхолдерами**. Оператор редактирует прозу; структурную динамику (список позиций, дата) система подставляет по меткам. Диалог превью (ч.2) остаётся как разовая правка поверх отрендеренного шаблона.

Заодно — **нераскрытие тендера** (решение оператора 2026-07-09): из письма убираются номер тендера и ссылка на объявление, чтобы недобросовестный поставщик не пошёл участвовать вместо дачи КП (детали §5).

## 2. Границы

- **В ЭТОМ спеке:** сущность+таблица шаблона (на рынок), рендер плейсхолдеров в `KpEmailComposer` с фолбэком на текущий зашитый дефолт, API CRUD шаблона, страница «Шаблон письма КП» в UI.
- **НЕ здесь:** редактирование самого слова приветствия «Уважаемый(ая)» (остаётся авто по наличию ФИО); HTML-письмо; часть 3 «разбор ответов». Раскрытие тендера/заявки в письме — сознательно убрано (§5), плейсхолдеров `{{предмет}}`/`{{ссылка}}` нет.

### Что НЕ трогаем (уже готово)
- Диалог превью + `subjectOverride`/`bodyOverride` в `/send` (ч.2) — разовая правка поверх шаблона.
- Серверный токен `[КП-id]` (`KpToken.subjectToken`) — в шаблон НЕ входит, приклеивается в `compose`.
- `EmailService`, `MailReceiveService`, приём ответов.

## 3. Согласованные решения (brainstorming)

| Решение | Выбор |
|---|---|
| Что редактируем | текст + плейсхолдеры (структурные блоки — авто) |
| Область | **отдельный шаблон на рынок** (KZ / RF) |
| Тема | тоже редактируемая шаблоном (токен `[КП-id]` всё равно серверный) |
| `{{приветствие}}` | авто (система решает «Уважаемый(ая) …»/«Здравствуйте!» по ФИО) |
| Кнопка сброса | есть, называется **«Сбросить»** (вернуть зашитый дефолт) |

## 4. Модель данных (миграция V10)

Новая сущность `EmailTemplate` (`entity/`), общая (НЕ `MarketScoped` — читаем по явному `market`, не через `MarketContext`-аспект):
```
CREATE TABLE email_template (
  id               BIGSERIAL PRIMARY KEY,
  market           VARCHAR(2) NOT NULL UNIQUE,   -- 'KZ' | 'RF'
  subject_template TEXT NOT NULL,
  body_template    TEXT NOT NULL,
  updated_at       TIMESTAMPTZ
);
```
Сущность: Lombok, `@Enumerated(STRING) Market market` (UK), `subjectTemplate`, `bodyTemplate`, `OffsetDateTime updatedAt`. Репозиторий `EmailTemplateRepository extends JpaRepository<EmailTemplate, Long> { Optional<EmailTemplate> findByMarket(Market m); }`.

**Сид V10:** по одной строке на KZ и RF с дефолтным текстом (см. §6; рыночно-нейтральный, различия — через плейсхолдеры, поэтому текст одинаков для обеих строк), чтобы редактор сразу показывал осмысленное. Дефолт хранится и в коде (единственный источник для «Сбросить» и фолбэка) — сид совпадает с ним.

## 5. Плейсхолдеры

Метки `{{...}}`, регистронезависимо не требуется (фиксируем нижний регистр русских слов). Система подставляет:

| Плейсхолдер | Значение | Источник |
|---|---|---|
| `{{приветствие}}` | «Уважаемый(ая) <Фамилия Имя>!» если есть ФИО, иначе «Здравствуйте!» | `pr.distributor` |
| `{{компания}}` | «ТОО «West-Med»» / «ООО «РЕГИОН-МЕД»» | `market.companyShortName()` |
| `{{позиции}}` | многострочный список: «— Лот N: <модель/лот> (<бренд>), РУ № … — N шт.\n  Требования (из ТЗ): …» | `pr.items` (структурный блок) |
| `{{дедлайн}}` | «Просим ответить до 03.07.2026.» или пусто | `tender.deadline` |
| `{{реестр}}` | «НЦЭЛС РК» / «Росздравнадзора» | `market` |

- Тема — тоже шаблон (короткий, дефолт «Запрос коммерческого предложения»). В `compose` перед человеческой темой всегда приклеивается `KpToken.subjectToken(pr.getId())` (это НАШ внутренний id КП, он не раскрывает тендер).
- Неизвестная метка `{{xyz}}` — оставляется как есть (не падаем); лог debug.
- **⚠ Письмо НЕ раскрывает конкретный тендер/закупку** (решение оператора 2026-07-09): недобросовестный поставщик, узнав номер объявления, пойдёт участвовать сам вместо дачи КП. Поэтому **нет плейсхолдеров** `{{предмет}}` (номер тендера/заявки) и `{{ссылка}}` (ссылка на объявление goszakup/zakupki) — они убраны и из письма, и из набора меток. Письмо просит КП «по следующим позициям» без привязки к закупке. `{{дедлайн}}` — нейтральный «просим ответить до …» (дата сама по себе тендер не выдаёт). Если поставщик спросит контекст — оператор отвечает вручную.

## 6. Рендер и фолбэк (`KpEmailComposer` + новый `EmailTemplateRenderer`)

- Новый компонент `EmailTemplateRenderer` (util/service): `render(String template, Map<String,String> vars)` — заменяет `{{key}}` на значение; отсутствующие ключи не трогает.
- `KpEmailComposer`:
  - грузит шаблон рынка через `EmailTemplateRepository.findByMarket(market)`;
  - если строки нет ИЛИ поле пустое → берёт **зашитый дефолт** (константы `DEFAULT_SUBJECT`, `DEFAULT_BODY` — вынести текущий текст `buildBody`/`humanSubject` в шаблонные строки с плейсхолдерами). Дефолт **рыночно-нейтрален**: все различия рынков теперь через плейсхолдеры (`{{компания}}`/`{{реестр}}`), поэтому одна константа темы и одна тела; сид V10 кладёт их ОДИНАКОВО в строки KZ и RF (дальше оператор правит каждую независимо);
  - строит `Map` плейсхолдеров (в т.ч. многострочный `{{позиции}}` тем же кодом, что сейчас в цикле) → `renderer.render(...)`.
  - `compose`: тема = `subjectToken(id) + " " + render(subjectTemplate)`; тело = `render(bodyTemplate)`. `composeForPreview`: тема без токена.
- **Инвариант:** при пустой таблице вывод рендерится из зашитого дефолта. Дефолт — те же блоки в том же порядке, что и сейчас, МИНУС раскрытие тендера: вступление «просит предоставить коммерческое предложение **по следующим позициям**:» (без номера тендера), строка объявления/ссылки удалена, дедлайн переформулирован в «Просим ответить до …». Дефолтный текст = единственный источник (сид V10, фолбэк, «Сбросить» ссылаются на одни и те же константы).
- **Изменение поведения существующего письма (ч.2):** сейчас композер вставляет номер тендера в тему/тело и ссылку на объявление — это УБИРАЕТСЯ по §5. Метод `announceLink` в `KpEmailComposer` становится мёртвым → удалить. Существующие тесты композера, пиновавшие «по тендеру № …» в теме и `https://goszakup…` в теле, **обновляются** под новый нераскрывающий дефолт — заложено спецификой.
- Зависимость `KpEmailComposer` на репозиторий — конструкторная; сеть/БД-чтение под tx/OSIV как у прочих (композер зовётся из `dispatch` после коммита, сессия OSIV открыта на HTTP-потоке; для фонового приёма композер не используется).

## 7. API

- `GET /api/email-template?market=KZ` → `{ market, subject, body, updatedAt }` — текущий сохранённый (или дефолт, если строки ещё нет).
- `PUT /api/email-template` body `{ market, subject, body }` → сохранить (upsert по market), `@PreAuthorize("hasRole('ADMIN')")`, `updatedAt=now`.
- `GET /api/email-template/default?market=KZ` → `{ subject, body }` зашитого дефолта (для кнопки «Сбросить» — фронт подставляет в поля, не сохраняя до нажатия «Сохранить»).
- Валидация PUT: тело не пустое; предупреждение (не ошибка) если в `body` нет `{{позиции}}` — вернуть флаг `warnings: ["no-positions"]`, фронт покажет мягкое предупреждение, но сохранит.

## 8. Фронт (новая страница `email-template`, группа «Система»)

- Роут `/email-template` под `LayoutComponent`/`authGuard`; ссылка «Шаблон письма КП» в группе «Система» сайдбара (рядом с Отчёты/Пользователи/О системе).
- Standalone-компонент: заголовок с указанием активного рынка («Шаблон для: West-Med (KZ)»); поле «Тема» (input) + «Текст письма» (textarea, крупная); панель-легенда доступных плейсхолдеров с описанием — клик по метке вставляет её в последнее активное поле (тема/тело); кнопки **«Сохранить»** и **«Сбросить»**.
- Загрузка: `GET /api/email-template?market=<active>`; смена рынка (глобальный переключатель перезагружает страницу — покажет шаблон нового рынка).
- «Сбросить» → `GET .../default` → заполняет поля дефолтом (тост «Загружен стандартный шаблон — нажмите Сохранить, чтобы применить»), НЕ сохраняет молча.
- Мягкое предупреждение при сохранении без `{{позиции}}` (из `warnings`).
- Паттерны проекта: `ApiService`, `NotificationService`, `cdr.detectChanges()`.

## 9. Тестирование

- `EmailTemplateRendererTest` — подстановка каждого плейсхолдера; отсутствующий ключ остаётся; многострочные `{{позиции}}`.
- `KpEmailComposerTest` (расширить + обновить) — с шаблоном из БД рендерит его; с пустой таблицей рендерит зашитый дефолт. **Существующие ассерты, которые ТЕПЕРЬ падают и обновляются** (нераскрытие тендера): тема больше НЕ содержит «по тендеру № 17276387-1» (станет «Запрос коммерческого предложения»); тело больше НЕ содержит `https://goszakup.gov.kz/...` (строка ссылки удалена). Сохраняются ассерты блоков: приветствие, компания, позиции/РУ/ТЗ, дедлайн (новая формулировка «Просим ответить до …»), реестр. Новые ассерты: тема несёт серверный токен `[КП-id]`; `composeForPreview` без токена; шаблон из БД перекрывает дефолт; письмо НЕ содержит номер тендера и ссылку на объявление (анти-лик проверка).
- `EmailTemplate` API/сервис — GET (сохранённый и дефолтный), PUT upsert + `warnings`-флаг без `{{позиции}}`, гард ADMIN.
- Гейт «зелёного»: только 2 пред-существующих `ApplyAutoFillServiceTest`. Фронт — `npm run build`.
- Живая проверка (Playwright + MailHog): открыть «Шаблон письма КП» (KZ), изменить текст (напр. вставить свою фразу), «Сохранить» → отправить КП по тендеру → в MailHog письмо с новым текстом; «Сбросить» возвращает дефолт в поля.
